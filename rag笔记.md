# 第 1 课：RAG 是为了解决什么问题

## 1.1 这一课先记住一句话

> RAG 就是：先帮大模型从资料库里找资料，再让大模型根据资料回答问题。

它不是让大模型凭空回答，也不是简单搜索后把链接扔给你。

它的目标是：

1. 找到相关资料。
2. 理解资料内容。
3. 组织成自然语言答案。
4. 告诉你答案来自哪里。
5. 如果资料不够，就不要乱编。

---

## 1.2 原文在讲什么场景

英文材料讲的是一个大型企业内部知识助手。

你可以把它理解成：

> 公司内部版的“智能搜索 + 问答助手”。

假设一个大公司有：

- 500 多个工程团队。
- 5000 多名工程师。
- 200 万以上内部文档。
- 每天 1000 万次查询。

这些文档分散在很多地方：

| 来源 | 里面有什么 | 麻烦在哪里 |
|---|---|---|
| Confluence | 设计文档、操作手册、架构决策 | 页面多、层级深、很多过期内容 |
| GitHub | README、代码注释、PR 讨论 | 代码和文字混在一起，变化很快 |
| Slack | 群聊、事故讨论、临时决策 | 很口语、很杂、上下文分散 |
| Google Docs | RFC、需求文档、会议纪要 | 权限复杂、版本多 |
| 内部 Wiki / S3 | PDF、图表、老系统文档 | 格式乱、元数据少 |

大公司的问题不是没有知识，而是：

> 知识到处都是，但你找不到。

比如一个工程师可能想问：

- 支付服务怎么处理重试？
- 某个接口的鉴权方式是什么？
- 上次 Q3 迁移之后 checkout 为什么变慢？
- 某个事故当时是怎么解决的？

答案可能在 Confluence，也可能在 GitHub README，也可能在 Slack 的某个讨论串里。

---

## 1.3 为什么不用“超长上下文”直接解决

你可能会想：

> 现在大模型上下文不是很长吗？把所有文档都塞给它不就行了吗？

原文说：不行。

原因是规模差太多。

假设：

- 公司有 200 万篇文档。
- 每篇文档平均 2000 个 token。
- 总共就是 40 亿 token。

而大模型上下文窗口就算有 128K、200K token，也远远装不下 40 亿 token。

这就像：

> 你问图书管理员一个问题，不可能让他每次把整个图书馆所有书都搬到你面前再回答。

而且就算能搬过来，也会有两个问题：

1. **太慢**：每次都读所有文档，延迟很高。
2. **太贵**：大模型按输入输出 token 收费，所有文档都塞进去成本爆炸。

所以结论是：

> 长上下文可以帮助阅读少量资料，但不能解决海量知识库搜索问题。

---

## 1.4 为什么不用普通关键词搜索

第二个想法是：

> 那我用 Elasticsearch 这种关键词搜索不就行了吗？

也不够。

关键词搜索的问题是：它看的是“字面匹配”，不是真的理解你的意思。

例子：

你问：

> payments service 怎么做 auth？

也就是：“支付服务怎么做鉴权？”

但相关文档标题可能叫：

> OAuth2 Integration Guide for Payment Gateway

关键词搜索可能匹配不好，因为：

- 你说 auth，文档写 OAuth2。
- 你说 payments service，文档写 Payment Gateway。
- 你问的是“怎么做”，文档标题是“集成指南”。

人能看懂它们相关，但关键词搜索不一定能。

所以结论是：

> 关键词搜索擅长找精确词，比如函数名、接口名、错误码；但它不擅长理解用户意图。

---

## 1.5 RAG 到底解决什么

RAG 结合了两件事：

| 能力 | 作用 |
|---|---|
| 检索 Retrieval | 先从大量文档里找到相关片段 |
| 生成 Generation | 再让大模型基于这些片段组织答案 |

也就是说，RAG 不要求大模型记住所有公司知识。

它的流程更像这样：

1. 用户问问题。
2. 系统先去资料库里搜索相关内容。
3. 系统把最相关的几段资料交给大模型。
4. 大模型根据这些资料回答。
5. 答案里标明来源。

类比一下：

> 普通大模型像“凭记忆答题的学生”。  
> RAG 像“开卷考试的学生”，先翻资料，再组织答案。  
> 好的 RAG 系统还要求学生必须标注答案来自哪一页。

---

## 1.6 一次 RAG 查询是怎么跑完的

原文用的问题是：

> How does the payments service handle retries?

中文就是：

> 支付服务是怎么处理重试的？

一次普通 RAG 查询大概分 4 步。

---

### 第 1 步：理解用户问题

系统先看用户到底问什么。

它会识别：

- 关键对象：payments service，也就是支付服务。
- 关键动作：handle retries，也就是处理重试。
- 问题类型：这是一个事实 / 机制类问题。
- 复杂度：这个问题大概率一次搜索就能回答。

这一步可以理解为：

> 先把用户的问题翻译成更适合机器搜索的形式。

---

### 第 2 步：去资料库检索相关片段

系统会去找相关资料。

这里重点来了：RAG 检索的通常不是整篇文档，而是文档里的小片段。

这些小片段叫 **chunk**。

为什么不直接搜整篇文档？

因为一篇文档可能很长，里面只有一小段真正回答问题。

例如一篇《Payment Error Handling》文档里，只有某一段写了：

- 支付服务使用指数退避。
- 每次重试之间加入 jitter。
- 最多重试 3 次。
- 连续失败后打开 circuit breaker。

RAG 需要找的是这一段，而不是整篇文档。

---

### 第 3 步：合并和重排搜索结果

生产级 RAG 通常不会只用一种搜索。

它会同时用：

| 搜索方式 | 中文解释 | 优点 | 缺点 |
|---|---|---|---|
| 向量搜索 | 按语义相似度找内容 | 能理解相似意思 | 可能漏掉精确词 |
| 关键词搜索 | 按关键词匹配 | 找 API、函数名、错误码很准 | 不太理解同义表达 |

这叫 **Hybrid Retrieval，混合检索**。

举个例子：

- 用户问“鉴权怎么做”。
- 向量搜索能找到“OAuth2 集成指南”。
- 关键词搜索能找到具体的 `auth_token`、`401`、`payment-gateway`。

两种结果会合并起来。

然后系统还会做 **Re-rank，重排**。

重排的意思是：

> 初步找出 50 个可能相关的片段，再更仔细地判断哪个最相关，最后选出前 3-10 个给大模型。

为什么要重排？

因为第一轮搜索追求“别漏掉”，第二轮重排追求“选得准”。

---

### 第 4 步：大模型基于资料生成答案

最后，系统把选出来的片段放进 prompt，要求大模型：

- 只能根据提供的资料回答。
- 不要编造资料里没有的内容。
- 答案要标注来源。
- 如果资料不够，就说不知道。

比如最终回答可能是：

> 支付服务使用指数退避和 jitter 处理下游重试。每个服务都有重试预算，支付服务最多重试 3 次，基础延迟为 2 秒。连续 5 次失败后，熔断器会打开，防止重试风暴继续扩散。

在真实系统里，答案后面还会带来源，例如：

- 来源：Payment Error Handling
- 来源：payments-service README
- 来源：Resilience Patterns

这就是 grounding 和 citation。

| 词 | 中文理解 |
|---|---|
| Grounding | 答案必须基于资料，有依据 |
| Citation | 答案要标注来源 |

---

## 1.7 简单问题和复杂问题不一样

简单问题：

> 支付服务怎么处理重试？

一般一次搜索就够。

流程是：

> 问题 → 搜索 → 找到片段 → 大模型回答

复杂问题：

> checkout 为什么变慢？Q3 迁移之后发生了什么变化？

这个问题就麻烦很多。

因为它其实包含多个子问题：

1. checkout 慢在哪里？
2. Q3 迁移改了哪些东西？
3. 迁移前后有没有性能指标变化？
4. 有没有事故记录或 Slack 讨论？
5. 慢和迁移之间有没有因果关系？

这类问题一次搜索通常不够。

系统需要：

1. 先判断这是复杂问题。
2. 把它拆成多个子问题。
3. 分别搜索。
4. 看搜索结果够不够。
5. 如果不够，继续搜索。
6. 最后综合多个来源生成答案。

这就是 **Agentic RAG**。

你可以这样理解：

| 类型 | 像什么 |
|---|---|
| 普通 RAG | 搜一下资料，然后回答 |
| Agentic RAG | 像研究员一样，先拆问题，再多轮查资料，最后总结 |

## 1.8 这一课的核心概念

### RAG

检索增强生成。先找资料，再让大模型基于资料回答。

### Retrieval

检索。负责从知识库中找到相关文档片段。

### Generation

生成。负责把检索到的资料组织成自然语言答案。

### Chunk

文档切出来的小片段。RAG 通常检索 chunk，而不是整篇文档。

### Embedding

把文本转换成向量，也就是一组数字，让机器可以计算“意思像不像”。

### 向量检索

根据语义相似度找内容。适合找“意思相近”的资料。

### 关键词检索

根据关键词匹配找内容。适合找 API 名称、函数名、错误码、专有名词。

### 混合检索

同时使用向量检索和关键词检索。

### Re-rank

重排。先粗略找一批候选，再精细排序，挑最好的内容给大模型。

### Grounding

让回答有依据，只能来自检索到的资料。

### Citation

引用来源。告诉用户答案来自哪篇文档、哪一段资料。

### Abstention

证据不足时拒答或说明不确定，不要乱编。

---

## 1.9 这一课你只要掌握这张图

```text
用户问题
  ↓
理解问题：识别意图、实体、复杂度
  ↓
检索资料：向量搜索 + 关键词搜索
  ↓
合并重排：挑出最相关的 chunk
  ↓
组装上下文：把资料放进 prompt
  ↓
大模型回答：基于资料生成、有引用、不乱编
  ↓
返回答案
```

一句话版：

> RAG = 搜资料 + 选资料 + 喂给模型 + 带引用回答。

# 第 2 课：生产级 RAG 不是 Demo

## 2.1 这一课先记住一句话

> 生产级 RAG 的重点不是“模型能不能回答”，而是“能不能稳定、便宜、可信地回答”。

做一个 Demo 很容易：

- 接个大模型。
- 接个向量库。
- 扔几篇文档进去。
- 搜出来几段内容塞进 prompt。

这能跑，但离生产级系统还差很远。

因为一旦文档变成几百万篇、用户变成几千人、查询变成每天几百万到上千万次，真正的问题就不再只是“能不能答”，而是：

- 找到的资料对不对。
- 成本能不能撑住。
- 延迟能不能接受。
- 权限会不会泄漏。
- 文档更新后答案会不会过期。
- 模型找不到证据时会不会乱编。
- 你能不能知道系统哪里坏了。

---

## 2.2 Demo 和生产系统的本质区别

你可以把它理解成这样：

| Demo RAG | 生产级 RAG |
|---|---|
| 文档少，几十篇到几千篇 | 文档多，几十万到几百万篇 |
| 不太考虑权限 | 必须严格按用户权限过滤 |
| 只要能答出来就行 | 要求答案正确、可追溯、可监控 |
| 不太在意成本 | 每天海量请求，成本必须精算 |
| 不太在意更新延迟 | 文档更新后要尽快可搜到 |
| 问题答错了影响小 | 一次错误答案就可能让用户失去信任 |

所以原文一直在强调：

> 生产级 RAG，本质上是一个数据系统、检索系统、评估系统，再加一个 LLM。

LLM 当然重要，但不是全部。

---

## 2.3 第一条原则：检索质量决定答案质量

原文最重要的一句话是：

> The LLM can only work with what retrieval gives it.

中文就是：

> 大模型只能基于你检索给它的资料工作。

如果你搜错了资料，大模型再强也没法凭空把正确答案变出来。

这就是为什么原文说，做 RAG 时，应该把大量精力放在这些地方：

- 文档怎么切块。
- 用什么 embedding。
- 检索策略怎么设计。
- 重排怎么做。
- 评估怎么验证检索真的有效。

一句话理解：

> 你喂给模型什么，它大概率就会围绕什么回答。

所以生产里常见的真实情况是：

- 检索很好 + 模型一般：答案通常还不错。
- 检索很差 + 模型很强：答案依然容易“自信地错”。

---

## 2.4 第二条原则：chunk 质量比 chunk 数量更重要

很多人第一次做 RAG，会觉得：

> 多塞一点上下文，总比少塞一点好吧？

原文明确反对这种想法。

原因很简单：

- 相关的 chunk 太少，模型没有证据。
- 不相关的 chunk 太多，模型会被噪声干扰。
- 上下文太长，还会增加成本和延迟。
- 中间部分的信息，模型可能反而注意不到。

原文提到一个很关键的现象，叫：

> lost in the middle

意思是：

> 当上下文很长时，模型容易忽略中间的信息。

所以生产系统追求的不是“塞得多”，而是：

> 塞得准。

五个高质量 chunk，往往比二十个一般质量 chunk 更有价值。

这也是为什么后面整份材料会把大量篇幅放在 chunking 上。因为 chunk 不是随便切，它直接决定检索效果。

---

## 2.5 第三条原则：不是所有问题都值得用最贵的模型

原文强调：

> cost-aware model routing is not optional

意思是：

> 按问题复杂度做模型路由，不是优化项，而是必需项。

原因很现实。一个企业每天如果有上千万次查询，LLM 推理费用会是总成本里的大头。

有些问题很简单，比如：

- 用户服务的 endpoint 是什么？
- 4032 错误码是什么意思？
- 某个配置项放在哪个文件？

这种问题没必要每次都调用最贵、最慢的大模型。小模型通常就够了，而且更便宜、更快。

只有这些问题才更值得路由到大模型：

- 需要比较。
- 需要多步推理。
- 需要跨多个资料来源整合。
- 需要处理复杂上下文。

你可以把模型路由理解成：

| 问题类型 | 更适合的模型 |
|---|---|
| 简单查找题 | 小模型 |
| 多跳分析题 | 大模型 |

生产系统的目标不是“所有请求都用最强模型”，而是：

> 用刚刚够用的模型，把质量、延迟、成本一起平衡好。

---

## 2.6 第四条原则：系统必须敢于说“不知道”

原文有一句很重的话：

> Fail safe, not fail silent.

意思是：

> 当系统没把握时，宁可明确说不确定，也不要悄悄编一个看起来很像真的答案。

为什么？

因为在企业内部知识场景里，错误答案的破坏性很强：

- 工程师可能按照错误 runbook 处理事故。
- 错误接口说明会让人接错服务。
- 错误架构信息会误导后续设计。

相比之下，一个诚实的回答更可接受：

> 我目前没有足够证据给出确定答案，下面是最相关的资料来源。

这其实是在保护用户信任。

所以一个好的 RAG 系统，不只是“能答”，还要“知道什么时候不该答满”。

---

## 2.7 第五条原则：评估和观测不是可选项

很多 RAG Demo 最大的问题是：

> 看起来挺聪明，但你不知道它到底是不是稳定地对。

原文强调两件事必须长期存在：

1. 持续评估。
2. 全链路观测。

持续评估的意思是：

- 不是做完一次 demo 测几个问题就结束。
- 每次改 chunking、embedding、检索参数、prompt，都应该重新评估。
- 既要有离线评估，也要有线上反馈。

全链路观测的意思是：

- 你要知道检索慢在哪里。
- 你要知道 re-rank 效果怎么样。
- 你要知道哪个模型花钱最多。
- 你要知道哪种问题类型最容易答错。

如果没有这些指标，你遇到问题时只会看到一个现象：

> 用户觉得不好用。

但你不知道到底是检索坏了、模型坏了、权限错了，还是缓存过期了。

---

## 2.8 第六条原则：文档新鲜度会直接影响信任

原文明确说：

> Stale answers erode trust faster than wrong answers.

可以理解为：

> 过期答案，对用户信任的伤害非常大。

为什么？

因为工程师通常默认“内部知识系统说的应该是最新的”。如果系统给出两周前的旧文档结论，用户就会很快不信它。

所以生产级 RAG 必须考虑：

- 文档新增后多久能被搜到。
- 文档更新后多久能替换旧版本。
- webhook 丢失时怎么补偿。
- 是否有轮询和对账机制。

这也是为什么原文把 freshness 当成硬要求，而不是锦上添花。

---

## 2.9 常见反模式

这一节可以当成避坑清单。

原文提到几个非常典型的错误做法：

### 1. 整篇文档只做一个向量

问题在于粒度太粗。

用户问的是文档中的一个小点，但你拿整篇文档做语义表示，召回精度会很差。

### 2. 所有文档都用同一种切块方式

问题在于文档类型差异太大：

- Slack 讨论串和 API 文档不一样。
- RFC 和代码文件不一样。
- Runbook 和 PR 描述也不一样。

一刀切会让很多 chunk 天生质量差。

### 3. 指望模型自己发现幻觉

不行。模型非常擅长把“像真的”话说得很顺。

如果没有 grounding、citation 和后续校验，它完全可能引用一个并不存在的接口或流程。

### 4. 没有评估体系

只看 demo 感觉，是最危险的错觉。

没有 golden dataset、没有反馈回路、没有线上采样评估，就不知道系统是在进步还是退步。

---

## 2.10 技术选型在解决什么问题

原文第 5 节不是让你背厂商名单，而是让你理解：

> 每个组件都是在解决某一类具体问题。

可以先看这个简化版映射：

| 组件层 | 作用 | 原文推荐思路 |
|---|---|---|
| 文档解析 | 把不同来源文档转成统一结构 | 通用解析器 + 自定义解析逻辑 |
| 切块引擎 | 按文档类型切出高质量 chunk | 多策略切块，不是一刀切 |
| Embedding 模型 | 把文本变成可做语义搜索的向量 | 质量、维度、成本一起权衡 |
| 向量库 | 存 chunk 向量并做相似度搜索 | 重点看检索速度、过滤能力、扩展性 |
| 关键词索引 | 做 BM25 精确检索 | 适合专有名词、路径、错误码 |
| Re-ranker | 对候选结果重新排序 | 提高最终送给模型的上下文质量 |
| LLM | 负责生成最终答案 | 按复杂度路由不同模型 |
| 元数据库 | 记录权限、元数据、反馈 | 事务一致性很重要 |
| 缓存 | 处理重复查询 | 直接节省成本和延迟 |
| 队列 | 解耦 ingestion 和 embedding | 支持重试、回放、削峰 |
| 观测与评估 | 发现问题和持续优化 | 不是附属品，是主系统一部分 |

这节你不用强记具体产品名，先抓住选型逻辑。

---

## 2.11 一个关键认知：模型不是必须同一家

原文特别提醒了一个容易误解的点：

> chunking 模型、embedding 模型、生成模型，可以来自完全不同的供应商。

也就是说，你完全可以：

- 用一种方案做 chunking。
- 用另一种 embedding API 做向量化。
- 用第三种大模型做最终生成。

它们不需要来自同一家。

真正强耦合的地方只有一个：

> 入库时用的 embedding 模型，必须和查询时用的 embedding 模型一致。

因为如果文档向量和查询向量不在同一个向量空间里，相似度就没有意义。

这句话后面学 embedding 时会非常重要。

---

## 2.12 为什么这里选 RAG，而不是微调或长上下文

原文把三种方案放在一起比较：

| 方案 | 更适合什么场景 | 主要问题 |
|---|---|---|
| RAG | 大规模、频繁更新的知识库 | 需要做好检索和工程系统 |
| Fine-tuning 微调 | 统一语气、术语、格式 | 知识更新慢，容易过期 |
| Long Context 长上下文 | 小规模材料分析 | 规模一大，成本和效果都会变差 |

为什么这份材料里的场景适合 RAG？

因为它满足三个条件：

1. 文档太多，长上下文装不下。
2. 文档更新太快，微调跟不上。
3. 需要引用来源，必须知道答案依据的是哪篇文档。

所以这里的结论不是“RAG 最强”，而是：

> 面对大规模、变化快、要求可追溯的企业知识库，RAG 最合适。

---

## 2.13 规模估算：为什么原文一直强调数字

从这一节开始，材料开始进入“系统设计”视角。

因为生产系统不能只会讲概念，还得能粗略估算资源。

原文给出的基础设定是：

- 文档总数：200 万。
- 平均每篇文档约 5 个 chunk。
- 总 chunk 数：约 1000 万。

这三个数字非常关键，因为后面几乎所有容量估算都从这里展开。

---

## 2.14 存储怎么估

原文给了一个直观估算：

### 向量库存储

如果有 1000 万个 chunk，每个 chunk 一个 1024 维向量，每维 4 字节：

```text
10,000,000 × 1024 × 4 bytes ≈ 40 GB
```

这只是原始向量大小，还没算：

- 元数据。
- 索引结构。
- 额外开销。

所以真实的向量库存储会更大。原文给出的量级大约是：

> 18 GB 到 57 GB

差异取决于你有没有做量化压缩。

### 关键词索引

关键词索引不只是存原文，还要存倒排结构，所以体积通常会膨胀。原文给出的量级大约是：

> 54 GB

### 元数据库

除了文本和向量，还要存：

- 文档元数据。
- chunk 记录。
- ACL 权限表。
- 反馈记录。

原文给出的 PostgreSQL 量级大约是：

> 16 GB

这里你要学会的不是精确数字，而是思路：

> 做容量估算时，要把向量、索引、元数据、权限、索引开销分开算。

---

## 2.15 流量怎么估

原文假设每天 1000 万次查询。

平均 QPS 估算公式很简单：

```text
10,000,000 / 86,400 ≈ 115 QPS
```

但生产里不能只看平均值，因为平均值没什么意义。

真正影响系统设计的是：

- 高峰 QPS。
- 突发 QPS。

原文估算：

- 平均 QPS：约 115
- 高峰 QPS：约 350 到 500
- 突发 QPS：约 575

为什么会有高峰？

- 上午集中上班。
- 周一早上。
- 故障处理期间。
- 某个系统上线或回滚时。

所以系统设计时，你不能按“平均 115 QPS”买资源。你至少要能扛住峰值。

---

## 2.16 单次查询的时间花在哪里

原文把一条查询拆开估算，大概是这样：

| 阶段 | 大致耗时 |
|---|---:|
| 查询向量化 | 5-10ms |
| 向量搜索 | 10-50ms |
| BM25 搜索 | 5-20ms |
| Re-rank | 80-150ms |
| ACL 权限过滤 | 5-15ms |
| LLM 生成 | 400-1500ms |

结论非常清楚：

> 整体延迟里，最贵的通常不是搜索，而是最终的大模型生成。

所以原文说：

> 成本和延迟控制里，模型路由不是可选项。

如果所有问题都走大模型，钱会烧得很快，延迟也难压下来。

---

## 2.17 这一课你要记住的数字

不需要全背，但建议先记住这些量级：

| 指标 | 量级 |
|---|---|
| 文档数 | 200 万 |
| chunk 数 | 1000 万 |
| 日查询量 | 1000 万 |
| 平均 QPS | 115 |
| 峰值 QPS | 350-500 |
| 向量原始存储 | 约 40 GB |
| 单次查询总延迟 | 常见在 1 到 5 秒量级 |

这些数字的作用不是考试，而是帮你建立系统感觉：

> 这已经不是一个“玩具问答工具”，而是一个真正的在线生产系统。

---

## 2.18 这一课的核心结论

第 2 课学完，至少要建立这几个判断：

1. RAG 进入生产后，重点不是“能不能答”，而是“答得准不准、快不快、稳不稳、贵不贵”。
2. 检索质量决定上限，大模型不能凭空修正错误检索。
3. chunk 质量比 chunk 数量更重要，乱塞上下文会拉低效果。
4. 模型路由必须做，否则成本和延迟都扛不住。
5. 评估、观测、新鲜度、权限控制，都是主系统的一部分。

---

# 第 3 课：文档是怎么进入 RAG 系统的

## 3.1 这一课先记住一句话

> RAG 的答案质量，不只是检索和模型决定的，前面的文档摄取质量同样决定上限。

很多人看 RAG，只看到后半段：

- 用户提问。
- 检索资料。
- 模型生成答案。

但原文第 7 节强调的是前半段：

> 文档是怎么被抓进来、清洗好、变成标准内容、再送去切块和向量化的。

如果前面这一步做差了，后面再强也救不回来。

---

## 3.2 什么叫 ingestion

`ingestion` 可以先理解成：

> 把各种来源的原始文档，变成 RAG 系统能处理的标准输入。

原始文档本来很乱：

- 有的是 Confluence 页面。
- 有的是 GitHub README。
- 有的是 Slack 讨论串。
- 有的是 Google Docs。
- 有的是 PDF 或 S3 文件。

它们格式不同、权限不同、更新方式不同，不能直接塞进向量库。

所以 ingestion 要做的事情是：

1. 发现哪些文档新建了、更新了、删除了。
2. 把文档内容拉下来。
3. 提取正文和元数据。
4. 统一成内部标准格式。
5. 做去重。
6. 再送去后面的 chunking 和 embedding。

一句话：

> ingestion 是 RAG 的“进料系统”。

---

## 3.3 为什么原文说 ingestion 经常是事故源头

原文开头就提醒：

> connector 和 parsing 往往被低估，但生产里它们经常是问题高发区。

原因并不复杂，因为这部分要对接的外部系统太多，而且都不稳定：

- API 可能限流。
- webhook 可能丢事件。
- 文档格式可能变化。
- 权限可能同步不及时。
- PDF 可能抽取失败。
- Slack 讨论串可能上下文不完整。

你可以这样理解：

> ingestion 不是“搬文件”，而是在跟很多异构系统长期同步数据。

只要同步链路任何一环出问题，最终用户看到的答案就会过期、缺失或错误。

---

## 3.4 第一步：连接器在做什么

原文说，每个来源通常都要有一个单独的 connector service，也就是连接器服务。

不同来源的典型接法：

| 来源 | 常见接入方式 |
|---|---|
| Confluence | REST API + webhook |
| GitHub | Webhook + REST API |
| Slack | Events API + 历史消息接口 |
| Google Docs | Drive API + 推送通知 |
| S3 | 对象事件通知 |

虽然每种来源接法不同，但连接器的基本流程其实一样：

1. 检测变化。
2. 拉取文档。
3. 提取文本和元数据。
4. 把更新事件发布到消息队列。

原文里这里用的是 Kafka topic。

所以连接器不是直接把文档塞进向量库，而是先把“有文档变了”这件事，可靠地交给后面的处理流水线。

---

## 3.5 为什么不能只靠 webhook

这一点非常重要。很多初学者会觉得：

> 既然外部系统能发 webhook，收到事件就更新，不就够了吗？

原文明确说：不够。

因为 webhook 虽快，但不可靠。

现实里的问题包括：

- 某些系统会漏事件。
- 高峰期可能被限流。
- 重试机制不一定稳。
- 网络抖动会让你错过更新。

所以生产里常见做法不是 “webhook 或轮询”，而是：

> webhook + polling + reconciliation

也就是：

1. `webhook` 负责快。
2. `polling` 负责兜底。
3. `reconciliation` 负责对账补漏。

原文举的思路是：

- 平时靠 webhook 快速感知更新。
- 每天再跑一次对账任务。
- 重新拉取最近 24 小时修改过的文档。
- 和数据库里已有记录比较 hash。
- 发现漏掉的就重新入队。

你可以把它理解成：

> webhook 是前线侦察兵，对账任务是夜间清点库存。

生产系统不能只靠“应该能收到”，而要靠“即使漏了也能补回来”。

---

## 3.6 第二步：解析器在做什么

文档拉下来之后，不能直接进入后续流程。因为不同来源格式差异太大。

所以解析器要做的，是把它们变成统一的数据结构。

原文给了一个 `ParsedDocument` 的结构，核心字段大概包括：

| 字段 | 含义 |
|---|---|
| `doc_id` | 文档唯一标识 |
| `source` | 文档来自哪里 |
| `source_id` | 来源系统里的原始 ID |
| `title` | 标题 |
| `content` | 统一后的正文内容 |
| `content_type` | 文本、代码、混合、对话等类型 |
| `metadata` | 作者、时间、标签等元数据 |
| `permissions` | 谁能访问这篇文档 |
| `parent_doc_id` | 父文档或线程关系 |
| `url` | 回跳原文的链接 |
| `content_hash` | 内容哈希，用于检测变化 |

这里最重要的理解不是字段名，而是：

> 解析器的目标，是把“异构原文”变成“统一内部格式”。

只有这样，后面的 chunking、embedding、权限过滤才能一致工作。

---

## 3.7 不同来源文档，怎么统一处理

原文列了几种典型来源的解析思路：

### Confluence

- 原始格式可能是 XML 风格存储结构。
- 需要转换成 markdown 或其他统一文本结构。
- 要保留标题、表格、代码块这些有效信息。
- 纯展示性质的宏、样式噪声要去掉。

### GitHub

- README 和文档通常已经接近 markdown。
- 代码文件需要带语言标签。
- 最好补上文件路径上下文。
- PR 描述还可能包含 diff 摘要。

### Slack

- 真正有意义的单位通常不是一条消息，而是整个 thread。
- 要按时间顺序拼接。
- 需要保留是谁说的。
- emoji、reaction 这类噪声可以去掉。

### Google Docs

- 往往先导出成 HTML，再转成 markdown。
- 要保留标题、列表、表格。
- 视觉样式类信息通常不重要，可以去掉。

### PDF

- 解析难度通常最大。
- 要尽量保留版面结构。
- 太慢时还要有降级策略。

这部分的核心不是工具名，而是一个原则：

> 不同来源要尽量保留“对检索有价值的信息”，删除“只影响展示、不影响理解的信息”。

---

## 3.8 第三步：抽取元数据和权限

很多人一开始只盯着正文，其实不够。

原文里的 ingestion 还强调两类信息必须一起抽出来：

1. 元数据。
2. 权限信息。

元数据有什么用？

- 作者是谁。
- 最后更新时间是什么时候。
- 标签是什么。
- 来源系统是什么。
- 属于哪个团队。

这些信息后面可以用于：

- 过滤结果。
- 提升排序。
- 做 freshness 判断。
- 回跳源文档。

权限信息更关键。

因为企业内部知识系统有一个硬要求：

> 用户只能看到自己有权限看的文档。

所以权限不是最后再补的，而是在 ingestion 阶段就必须跟着文档一起进系统。

否则后面检索再准，也可能把不该看到的资料送出去。

---

## 3.9 第四步：为什么必须做去重

原文举了一个非常真实的例子：

同一份设计文档，可能同时出现在：

- Confluence 正文。
- Slack 讨论里引用。
- GitHub PR 描述里再次说明。

如果不去重，会发生什么？

- 搜索结果被重复内容占满。
- 模型上下文浪费在同一份信息上。
- 用户以为找到了多个证据，其实只是同一内容的重复副本。

所以原文推荐至少做两层判断：

### 完全重复

对标准化后的正文计算内容 hash。

如果 hash 一样，说明内容完全相同，可以直接跳过重复处理。

### 近似重复

有些文档只有一点格式差异，比如：

- 标题多了个空格。
- 表格样式不同。
- 少量装饰性文字不同。

这时可以做近似相似度判断，把高度相似的文档标出来进一步处理。

这部分的本质是：

> RAG 需要的是“信息增量”，不是“相同内容的多份副本”。

---

## 3.10 ingestion 的完整链路

把这一课内容串起来，完整流程大概是：

```text
外部系统文档发生变化
  ↓
连接器发现变化
  ↓
拉取原始文档
  ↓
解析正文、元数据、权限
  ↓
统一成内部标准格式
  ↓
做内容去重
  ↓
送入后续 chunking 和 embedding 流水线
  ↓
写入向量库、关键词索引、元数据库
```

要特别注意：

> ingestion 的输出，不是最终答案，而是“高质量、结构化、可检索的知识原料”。

---

## 3.11 这一课的核心结论

第 3 课学完，至少要建立这几个判断：

1. ingestion 是 RAG 的前置基础设施，不是可有可无的准备工作。
2. 连接器的职责是发现变化并可靠地把更新送入流水线。
3. webhook 只能解决“快”，不能单独解决“准”和“全”。
4. 解析器的核心价值是把异构文档统一成标准结构。
5. 元数据、权限、去重，都是 ingestion 的关键组成部分。

---

# 第 4 课：为什么切块决定 RAG 上限

## 4.1 这一课先记住一句话

> chunking 不是把文档“切开”这么简单，而是在决定检索系统以后能不能准确找到答案。

原文一开头就给了非常强的判断：

> 在 RAG 系统里，chunking 比大多数人想的更重要。

为什么？因为 RAG 检索的不是整篇文档，而是 chunk。

如果 chunk 切得不好，后面会出现两种典型问题：

- chunk 太小，信息不完整。
- chunk 太大，信息太杂，检索不准。

所以 chunking 的本质是：

> 在“上下文完整”与“检索精确”之间找平衡。

---

## 4.2 为什么 chunk 不能太小，也不能太大

原文给的权衡很直白：

### chunk 太小

会丢上下文。

比如一个 100 token 的片段里只写着：

> this approach has three advantages

问题是：

> this approach 到底指什么？

如果前面定义“this approach”的内容被切到另一个 chunk 里，这一小段本身几乎没法独立理解。

### chunk 太大

会让检索变脏。

比如一个 2000 token 的大 chunk，讲的是整个 auth 系统，但用户只问：

> token refresh 怎么做？

这时 embedding 会混进很多无关内容，真正和问题相关的信号就被稀释了。

所以你可以先记一个核心判断：

> chunking 的目标不是平均切块，而是让每个 chunk 尽量成为一个“可独立检索、可独立理解”的知识单元。

---

## 4.3 固定长度切块：最简单，但通常只是基线

最常见、也最容易上手的方式，是固定长度切块：

> 每 N 个 token 切一块，块和块之间再保留一点 overlap。

例如：

- `chunk_size = 512`
- `overlap = 50`

这种方法的优点很明显：

- 实现简单。
- 性能稳定。
- 对格式统一的文档能工作。

但它的问题也很明显：

- 可能从段落中间开始。
- 可能在句子中间结束。
- 可能把表格切断。
- 可能把一个完整概念拆成两半。

原文给了一个支付服务文档的例子，文档里有：

- `Retry Strategy`
- `Configuration`
- `Circuit Breaker`

如果用固定长度切块，可能把这些内容糊在一个大 chunk 里，或者更糟，刚好把表格切在中间。

如果用户问：

> `max_retries` 的默认值是多少？

这个 chunk 里会混入很多和问题无关的信息，导致 embedding 变得不够精准。

所以原文给固定长度切块的定位其实是：

> baseline，基线方案。

不是不能用，而是通常不够好。

---

## 4.4 递归切块：先尊重结构，再退化到长度

递归切块的思想是：

> 先按文档天然结构切，如果还太大，再继续往更细层级切。

原文里的层级大致是：

1. 先按二级标题切。
2. 再按三级标题切。
3. 再按段落切。
4. 再按行切。
5. 最后实在不行，再按句子或固定长度切。

这类方法为什么好？

因为它尽量保留了文档作者原本的组织方式。

举个例子，一篇 Confluence 文档本来就有：

- `## Retry Strategy`
- `### Configuration`
- `### Circuit Breaker`

递归切块后，就很自然得到：

- 一个介绍重试策略的 chunk。
- 一个专门讲配置表的 chunk。
- 一个专门讲熔断器的 chunk。

如果用户问参数默认值，`Configuration` 这个 chunk 会特别精准。

递归切块最适合什么？

- Confluence 页面
- GitHub README
- 技术设计文档
- 有清晰标题层级的规范文档

它什么时候会退化？

- 文档没有标题结构。
- Slack thread 这种天然扁平对话。
- 格式乱到几乎没有结构。

这时它就会慢慢退回到更普通的切法。

---

## 4.5 语义切块：不看标题，看意思什么时候变了

递归切块是按结构来切，语义切块则是按“话题变化”来切。

原理可以先用人话理解：

1. 先把文档拆成句子。
2. 给每个句子做 embedding。
3. 比较相邻句子的语义相似度。
4. 如果相似度突然下降，说明话题可能变了。
5. 在这里切一个边界。

它解决的是什么问题？

有些文档虽然很长，但没有清晰标题；或者即使有标题，真正的话题切换并不完全跟标题一致。

例如一篇设计文档里，可能自然地从：

- 问题背景
- 架构方案
- 实现细节

一路写下来，中间没那么规整。

这时候语义切块会比固定长度切块更聪明，因为它看的是“这一段到底还在不在讲同一个意思”。

它最适合：

- 长篇 prose 文档
- 多主题设计文档
- 会议纪要
- 结构不太规整的 Google Docs

它的代价是什么？

原文说，摄取时计算量会更高，因为你要给每个句子做 embedding，成本通常是普通切法的几倍。

但这个代价发生在 ingestion 阶段，不是每次查询都付，所以往往是可以接受的。

---

## 4.6 late chunking：先看完整文档，再生成 chunk 表示

这一节是整个 chunking 章节里最容易第一次读不懂的部分。

可以先抓住一句话：

> 传统做法是先切块再做 embedding，late chunking 是先让模型看完整体，再给块生成表示。

传统方式的问题是：

如果某个 chunk 里写着：

> this approach has three advantages

它本身并不知道 `this approach` 指的是前面哪一段定义的内容。

而 late chunking 的想法是：

1. 先把整篇文档送进长上下文 embedding 模型。
2. 模型在完整上下文里得到每个 token 的表示。
3. 然后再按 chunk 边界把这些 token 表示聚合成 chunk embedding。

这样得到的 chunk embedding，即使只对应一小段文本，也“带着整篇文档的上下文记忆”。

这特别适合什么文档？

- 大量代词引用前文的文档
- 强依赖前后文的技术规格说明
- 第 3 节老是引用第 1 节定义的文档

它的问题也很现实：

- 需要长上下文 embedding 模型。
- 文档太长时还是会超窗口。
- embedding 延迟更高。

所以它不是通用默认方案，而是：

> 为“上下文依赖很强”的文档付出更高代价，换更好的检索表达。

---

## 4.7 LLM-guided chunking：让模型直接参与切块决策

这一类方法更进一步：

> 不只是靠标题或相似度，而是让一个小模型来判断哪里应该切。

它通常会让模型输出：

- 每个 section 的开始和结束。
- 这个 section 的主题标签。
- 一句适合检索的摘要。

它适合什么场景？

- 结构混杂的复杂文档
- 图文混排、代码和分析交织的文档
- “人一看知道应该怎么分，但规则很难写死”的文档

它为什么贵？

因为你在 ingestion 阶段额外调用了一次 LLM。

如果文档量很大，这个成本会迅速累积。所以原文明确建议：

> 这种方法要有选择地用在高价值文档上，比如 RFC、设计文档，而不是全量默认开启。

一句话理解：

> LLM-guided chunking 追求的是“理解后再切”，但成本最高。

---

## 4.8 为什么没有一种切块策略能通吃所有文档

这是原文第 8 节最核心的结论之一：

> No single chunking strategy works for all document types.

中文就是：

> 没有任何一种切块策略，能对所有文档类型都最优。

原因很简单，因为文档类型差异太大：

- API 文档有明确层级。
- RFC 很长、跨多个话题。
- Runbook 是步骤导向。
- 代码文件天然有函数和类边界。
- Slack thread 的上下文靠整段对话累积。
- PDF 往往结构很差。

所以如果全都用一种切法，必然会让某些文档类型被错误处理。

这也是为什么生产系统最后会走向：

> 多策略 chunking pipeline。

---

## 4.9 生产级做法：按文档类型路由到不同切块策略

原文给了一个很典型的多策略映射，你可以先记这个思路：

| 文档类型 | 更适合的切法 | 原因 |
|---|---|---|
| API 文档 | 递归切块 | 标题结构清楚、段落自包含 |
| 设计文档 / RFC | 语义切块 + late chunking | 长文、跨主题、前后引用多 |
| Runbook / How-to | 按步骤递归切块 | 每一步通常是独立单元 |
| 代码文件 | 按函数 / 类边界切 | 逻辑边界天然存在 |
| Slack 讨论串 | 按 thread 保留整体 | 拆开会破坏上下文 |
| 会议纪要 | 更智能的语义或 LLM 辅助切块 | 内容松散、话题切换频繁 |
| PDF / 遗留文档 | 语义切块 | 结构差，只能更多依赖语义 |

这里真正重要的不是把表背下来，而是理解路由思想：

> 先判断文档是什么类型，再给它最合适的切块策略。

这样做的好处是：

- 结构化文档不被粗暴切断。
- 非结构化文档也不会被硬套标题规则。
- 整体检索质量更稳定。

---

## 4.10 chunk metadata 为什么重要

原文最后提到，每个 chunk 不只是正文，还应该带上一组丰富 metadata。

例如：

- 它属于哪个文档。
- 它在文档中的位置。
- 它用了哪种 chunking 策略。
- 它上面的标题是什么。
- 它来自哪个系统。
- 它的权限是什么。

为什么这些字段重要？

因为后面很多能力都靠它们：

- 检索结果要回跳原文。
- 需要做父子上下文扩展。
- 需要知道一个 chunk 属于哪个 section。
- 需要对不同策略效果做评估。
- 需要做权限过滤。

一句话：

> chunk 不是一段裸文本，而是“带结构信息的检索单元”。

---

## 4.11 chunking 的完整思路

把这一课内容串起来，你可以这样理解生产级 chunking：

```text
文档进入系统
  ↓
先判断文档类型
  ↓
为该类型选择最合适的切块策略
  ↓
生成尽量自包含、可理解、可检索的 chunk
  ↓
给每个 chunk 补充位置、标题、来源、权限等 metadata
  ↓
再送去做 embedding 和后续检索
```

所以 chunking 不是一个机械切分动作，而是：

> 在进入 embedding 之前，先把知识整理成适合搜索的最小有效单元。

---

## 4.12 这一课的核心结论

第 4 课学完，至少要建立这几个判断：

1. chunk 太小会丢上下文，太大会稀释信号，所以 chunking 本质是平衡问题。
2. 固定长度切块很常见，但通常只能作为基线。
3. 递归切块适合结构清晰文档，语义切块适合长篇多主题文档。
4. late chunking 和 LLM-guided chunking 是更高成本、针对特定问题的增强方案。
5. 生产级 RAG 最终通常会走向“按文档类型路由到多种切块策略”的方案。

---

# 第 5 课：文本为什么能变成可搜索的向量

## 5.1 这一课先记住一句话

> embedding 的作用，是把一段文本变成一个“可比较语义相似度”的数字表示。

RAG 能做语义检索，不是因为数据库突然理解了文字，而是因为文本先被变成了向量。

你可以先这样理解：

- 普通关键词搜索比的是“字面上有没有这个词”。
- embedding 检索比的是“意思上像不像”。

所以 embedding 是 RAG 里“理解语义”的基础设施。

---

## 5.2 embedding 到底在做什么

当一段文本进入 embedding 模型后，模型会输出一组数字，比如 1024 维或 3072 维向量。

这些数字本身人是看不懂的，但系统可以用它做一件非常重要的事：

> 比较两段文本在向量空间里离得近不近。

如果两个向量很接近，通常表示它们语义相似。

例如：

- “支付服务如何处理重试”
- “payments service retry policy”

字面不完全一样，但语义接近，所以 embedding 后的向量通常也会比较接近。

这就是为什么 RAG 可以找到“措辞不同但意思相关”的内容。

一句话：

> embedding 不是把文本压缩成编号，而是把“语义关系”映射到向量空间里。

---

## 5.3 为什么 embedding 是检索质量的地基

原文开头就强调：

> embedding model is the foundation of retrieval quality

意思是：

> embedding 模型选得差，后面整条检索链路都会一起吃亏。

因为后面的向量搜索、召回、重排，都是建立在“初始向量表示还算靠谱”的前提上。

如果 embedding 本身不能把语义关系表示好，就会出现这些问题：

- 相关内容离得不够近，搜不出来。
- 不相关内容反而被误召回。
- 后续 re-rank 需要处理更多噪声。
- 最终送给 LLM 的上下文质量下降。

所以第 5 课和第 4 课其实是连在一起的：

- 第 4 课解决“切什么文本单元去表示”。
- 第 5 课解决“这些文本单元怎么表示成可检索向量”。

---

## 5.4 embedding 模型怎么选

原文列了几种候选模型，但你现在不需要背型号，先抓选择维度。

选 embedding 模型时，核心要看这几件事：

| 维度 | 你在权衡什么 |
|---|---|
| 质量 | 语义表示准不准，检索效果好不好 |
| 延迟 | 生成向量快不快 |
| 成本 | API 成本或自部署成本是否能接受 |
| 维度 | 向量有多长，影响存储和检索成本 |
| 部署方式 | 用 API 还是自托管 |
| 能力特性 | 是否支持多语言、代码、降维等特性 |

原文在这个场景里选择 `text-embedding-3-large` 的原因，不是因为它“绝对最强”，而是因为它在下面几个点上平衡较好：

- API 使用方便。
- 延迟比较可控。
- 检索质量够好。
- 支持 Matryoshka 式降维。

所以这一节真正要学会的是：

> embedding 模型选择不是单看排行榜，而是看“质量、成本、延迟、存储”的综合平衡。

---

## 5.5 API 模型、自托管模型、轻量模型分别在解决什么问题

原文实际上给了三种思路：

### 1. API 模型

优点：

- 快速接入。
- 维护成本低。
- 通常稳定。

缺点：

- 长期规模大时，调用成本会累积。
- 受供应商额度和策略限制。

适合：

- 想先把系统稳定跑起来。
- 团队不想先投入 GPU 基础设施。

### 2. 自托管高质量模型

优点：

- 高吞吐场景下，长期成本可能更低。
- 可控性更强。
- 某些模型在特定 benchmark 上更强。

缺点：

- 要自己维护 GPU 服务。
- 部署和扩容复杂度更高。

适合：

- 文档量特别大。
- embedding 请求量特别高。
- 团队有基础设施能力。

### 3. 轻量模型

优点：

- 成本低。
- 资源要求小。
- 原型验证很方便。

缺点：

- 质量通常会差一些。

适合：

- 小规模系统。
- 原型期。
- 成本极度敏感的场景。

所以不是“哪一个最好”，而是：

> 你的规模、预算、团队能力不同，embedding 方案就会不同。

---

## 5.6 ingestion 阶段怎么批量做 embedding

原文第 9.2 节讲的是一个很实际的问题：

> 文档切成 chunk 之后，怎么高效地把它们批量转成向量。

典型流程是：

1. 新增或更新的 chunk 进入消息队列。
2. embedding worker 从队列里拉任务。
3. 把 chunk 按批次打包，比如 100 到 500 个一组。
4. 调 embedding API 或本地服务生成向量。
5. 向量写入向量库，文本写入关键词索引。
6. 两边都成功后，才确认这批任务完成。

这里为什么强调 batch？

因为 embedding 接口通常批量调用更划算：

- 吞吐更高。
- 单条请求开销更低。
- 更容易打满额度。

所以 embedding 不是“来一条 chunk 就单独请求一次”，而是一个批处理流水线。

---

## 5.7 一个硬规则：入库和查询必须用同一个 embedding 模型

这是原文第 9 节最重要的工程规则，没有之一：

> 文档入库时用的 embedding 模型，必须和查询时使用的 embedding 模型一致。

为什么？

因为向量相似度的前提是：

> 文档向量和查询向量必须在同一个向量空间里。

如果你以前用 `model A` 给 1000 万个 chunk 做了向量，现在查询时突然换成 `model B`，会发生什么？

- chunk 向量在空间 A。
- query 向量在空间 B。
- 你再去算 cosine similarity，数值就没有可比意义。

这不是“效果可能差一点”，而是：

> 语义空间不一致，检索逻辑本身就坏了。

你可以把它理解成：

> 用米做出来的尺子，不能直接跟英尺刻度混着量同一个东西。

chunking 模型和生成模型可以换供应商，但 embedding 这一点不能乱混。

---

## 5.8 embedding 模型升级，为什么这么麻烦

原文第 9.3 节讲的是 versioning 和 migration。

很多人第一次会想：

> 新模型更好，我把新查询直接改成用新模型不就行了吗？

不行。

因为旧文档向量还是按老模型生成的。只改查询模型会让查询空间和文档空间失配。

所以真正升级 embedding 模型时，你通常要做的是：

1. 建一套新的向量集合，比如 `chunks_v2`。
2. 在后台把全量文档重新做 embedding。
3. 新旧两套集合并存一段时间。
4. 用评估集验证新模型效果。
5. 验证通过后，再把查询流量切到新集合。
6. 保留旧集合一段时间，方便回滚。

这就是原文说的：

> blue-green re-indexing

它和应用发布里的蓝绿部署是同一个思路，只不过这里切换的是“向量数据版本”。

---

## 5.9 什么时候值得换 embedding 模型

原文的态度非常克制，不是“新模型一出就换”。

因为换 embedding 模型的代价很高：

- 要全量重建向量库。
- 需要重新跑评估。
- 可能引入质量回退。
- 还有切换和回滚成本。

所以真正值得换的情况通常是：

1. 质量有明显提升。
2. 成本能大幅下降。
3. 供应商要下线旧模型。
4. 你需要新的能力，比如多语言或代码检索。
5. 降维后能显著节省存储和延迟。

不值得急着换的情况通常是：

- 只提升一点点。
- 还没在你自己的数据上验证。
- 正在故障期或上线窗口。
- 没有蓝绿切换能力。

这一节真正要建立的意识是：

> embedding 模型升级是一次“数据层迁移”，不是简单改个配置。

---

## 5.10 为什么要在你自己的数据上评估

原文明确提醒：

> benchmark 分数高，不等于在你的语料上一定更好。

这点非常关键。

一个模型可能在通用评测集上比另一个高 3 分，但在你的企业文档上反而更差。因为你的语料可能有：

- 大量内部术语。
- 代码片段。
- 特定文档风格。
- 特定问题分布。

所以换模型前，原文建议做 shadow evaluation，大意是：

1. 先拿一部分语料做新模型向量。
2. 建一个影子集合。
3. 在 golden dataset 上跑检索评估。
4. 比较 `recall@10`、`MRR`、`NDCG@10`。
5. 只有新模型在你关心的指标上赢了或至少不差，才值得切。

这说明生产级 RAG 不是“谁家宣传强就用谁”，而是：

> 一切都要回到你自己的检索评估结果上。

---

## 5.11 降维和量化到底在省什么

原文最后一节讲的是压缩问题。

embedding 向量很占空间。比如：

- 3072 维向量
- 1000 万个 chunk

存储开销会非常大。

所以常见优化有两个：

### 1. 降维

例如把 3072 维缩到 1024 维。

如果模型支持这种做法，并且质量损失很小，那就能显著减少：

- 存储空间
- 内存占用
- 检索成本

### 2. 量化

例如把原本的 `float32` 向量压成 `int8`。

这会进一步减少内存占用，但通常会带来一定召回损失。

原文给出的核心思路不是让你背数字，而是：

> 先把检索质量做对，再考虑压缩；不要在检索本来就不够好的时候继续为省空间牺牲召回。

换句话说：

- 如果 recall 已经很边缘，就别急着压缩。
- 如果质量有余量，再用降维和量化换成本。

---

## 5.12 这一课的核心结论

第 5 课学完，至少要建立这几个判断：

1. embedding 的本质，是把文本映射到可比较语义相似度的向量空间。
2. embedding 模型是检索质量的地基，不能只看价格或排行榜。
3. 入库和查询必须使用同一个 embedding 模型，否则相似度没有意义。
4. embedding 模型升级本质上是一次向量数据迁移，通常需要蓝绿重建。
5. 降维和量化是成本优化手段，但前提是检索质量已经足够稳。

---

# 第 6 课：检索结果是怎么存起来并找出来的

## 6.1 这一课先记住一句话

> RAG 的检索层不是“查一个库”这么简单，而是多个存储系统一起配合，把相关内容找出来，再筛到足够准。

到第 5 课为止，我们已经有了：

- 标准化后的文档。
- 切好的 chunk。
- 每个 chunk 的 embedding。

接下来问题变成：

1. 这些数据该分别存在哪。
2. 用户提问时，怎么又快又准地把最相关内容找出来。

原文第 10-11 节回答的就是这件事。

---

## 6.2 为什么存储层不止一个数据库

很多人第一次做 RAG 会想：

> 我不是已经有向量库了吗？那不就够了？

原文给出的答案是：不够。

因为检索系统里其实有几种完全不同的数据需求：

- 需要做语义相似度搜索。
- 需要做精确关键词匹配。
- 需要做权限、审计、反馈等 SQL 查询。
- 需要做热点查询缓存。

所以生产系统通常不是一个库包打天下，而是：

| 存储层 | 主要职责 |
|---|---|
| 向量库 | 按语义相似度找 chunk |
| 关键词索引 | 按精确词匹配找 chunk |
| 元数据库 | 记录元数据、权限、血缘、反馈 |
| 缓存 | 直接复用近期相似问题的答案 |

一句话：

> 不同存储承担不同工作，组合起来才是完整检索系统。

---

## 6.3 向量库到底负责什么

向量库的任务很单纯：

> 根据 query embedding，从海量 chunk embedding 里找出最相似的一批。

原文这一节选的是 Qdrant，核心原因不是“它名字响”，而是它在这个场景里满足几件关键事：

- 能处理千万级向量。
- 支持分片和复制。
- 能调 HNSW 参数。
- 支持量化压缩。
- 支持按 metadata 做过滤。

这里你不用死记具体产品名，但要知道向量库里存的不是“答案”，而是这样的东西：

- 向量本身：表示语义。
- payload：正文、来源、文档 ID、权限、更新时间等信息。

也就是说，一个 chunk 存进去之后，不只是 1024 个数字，还带着它的上下文身份。

这一点非常重要，因为：

> 向量负责“像不像”，payload 负责“它是谁、能不能给用户看、怎么回跳原文”。

---

## 6.4 为什么还要关键词索引

向量搜索很强，但它不是万能的。

原文专门强调了几个它容易失手的场景：

- 错误码，比如 `ErrorCode 4032`
- 版本号
- 精确标识符
- 路径名
- 函数名

这种内容的关键，不在“语义”，而在“字面必须对得上”。

这就是 BM25 / Elasticsearch 这一层存在的原因。

你可以这样理解：

- 向量检索擅长找“意思相近”。
- BM25 擅长找“词完全命中”。

例如：

查询：

> `ErrorCode 4032`

BM25 很容易找到精确命中的 chunk。  
向量搜索则可能只找到“错误处理相关”的一般文档。

反过来，查询：

> how to handle failures gracefully

BM25 可能找不到原词匹配，但向量搜索能找到“Retry and Circuit Breaker Patterns”之类语义上很像的内容。

所以不是二选一，而是：

> 两种检索方式的盲点不同，放在一起更稳。

---

## 6.5 元数据库和缓存分别在解决什么问题

### PostgreSQL 这类元数据库

它不是用来做向量搜索的，而是用来做系统事实记录。

原文里它主要存这些东西：

- 文档元数据
- chunk 和文档的映射关系
- 权限 ACL
- 用户反馈
- 评估结果

为什么这些东西不全塞进向量库 payload？

因为很多后台分析、审计、管理查询，更适合 SQL：

- 某个团队最近有哪些文档同步失败？
- 哪些文档权限更新了？
- 哪类问题用户点踩最多？
- 哪种 chunk 策略效果更差？

向量库适合搜，元数据库适合查和管。

### Redis 这类语义缓存

缓存层解决的是：

> 同样或非常接近的问题，没必要每次都走完整条检索 + 生成链路。

原文这里做的是 semantic cache，不是纯字符串缓存。

也就是说，缓存命中的依据不是“问题文本一模一样”，而是：

> 新问题的 embedding 和历史问题 embedding 足够接近。

这样：

- “How does auth work?”
- “How does authentication work?”

虽然字符串不一样，但语义非常接近，就有机会命中缓存。

它的价值非常直接：

- 命中时延迟从秒级降到毫秒级。
- 节省检索成本。
- 节省 LLM 生成成本。

---

## 6.6 数据不是存进去就完了，还要维护生命周期

原文第 10.6 节讲的是一个很多教程不讲，但生产里很关键的问题：

> 向量和索引不是“建完就一直健康”，它们会老化、碎片化、残留脏数据。

典型问题包括：

- 文档删了，但旧 chunk 还在索引里。
- 更新后旧向量没有及时清掉。
- 软删除累计太多，影响性能。
- 索引碎片化，查询越来越慢。

所以生产里要做周期性维护：

- 清理 stale / orphan 数据。
- 对向量段做 compaction。
- 对关键词索引做合并优化。
- 给长期没更新的资料打 stale 风险标记。

一句话：

> 检索库也是生产数据库，要做持续运维，而不是一次性导入。

---

## 6.7 检索不是直接搜，先要理解用户问题

原文第 11.1 节强调：

> 用户原始问题往往并不适合直接拿去检索。

为什么？

因为用户的问题可能：

- 说得很模糊。
- 用了代词。
- 是上一轮对话的延续。
- 很短，但背后意图复杂。

所以检索前通常有一个 `query understanding` 层。

它会做几类事：

### 1. 分类

先判断问题属于哪类：

- 简单事实题
- how-to
- 分析题
- 多跳题
- 对话跟进题

因为不同问题，后面走的检索和模型路径可能不同。

### 2. 改写

例如用户说：

> that auth thing from last quarter

系统会把它改写成更适合检索的形式，比如：

> authentication changes implemented in Q3 2025

### 3. HyDE

对很抽象的问题，系统甚至会先假设性生成一段“可能的答案”，再用这段假设答案做 embedding 检索。

例如：

> Why is checkout slow?

直接做 embedding 可能太抽象；先生成一段关于 checkout latency 的假设说明，反而更接近真实文档语义。

这里你要建立的感觉是：

> 检索前的 query understanding，本质是在把“人类问题”翻译成“机器更容易搜准的形式”。

---

## 6.8 hybrid retrieval 为什么更稳

原文第 11.2 节的核心结论是：

> 向量检索和 BM25 的失败模式互补，所以混合检索通常优于单一路径。

生产系统的常见做法是：

1. 向量检索和 BM25 并行执行。
2. 各自返回一个候选列表。
3. 再把两个列表融合。

为什么比只走一个更稳？

因为：

- BM25 抓精确词很强，但不懂意图。
- 向量检索懂语义，但容易漏精确标识符。

把两路结果放一起，就同时兼顾了：

- lexical match
- semantic match

所以 hybrid retrieval 不是“为了复杂而复杂”，而是：

> 用两种不同视角看同一个问题，降低单一路径漏掉关键信息的概率。

---

## 6.9 RRF 和动态权重在解决什么问题

两个检索列表出来后，还得决定怎么合并。

原文介绍了两类典型思路：

### 1. RRF

Reciprocal Rank Fusion 的思路很简单：

> 一个文档如果同时在两边都排得靠前，就应该被明显加分。

它不要求你把不同模型的原始分数强行对齐，而是直接利用“排名位置”来融合。

所以 RRF 特别适合：

> 两条检索链的分数体系不完全一致，但你仍想稳定合并它们。

### 2. 动态 alpha

有些系统不是用 RRF，而是直接做加权融合。

例如：

```text
最终分数 = alpha × 向量分数 + (1 - alpha) × BM25 分数
```

关键在于 alpha 不一定固定。

例如：

- 概念类问题：更偏向向量检索。
- 错误码 / 代码标识符：更偏向 BM25。
- how-to：两边都需要。

所以这里真正的思想不是公式，而是：

> 不同问题类型，最优的检索融合权重可能不同。

---

## 6.10 为什么还要 re-rank 一次

这一步是很多人第一次做 RAG 时会省掉的，但原文把它放得很重要。

原因是：

> 第一轮检索追求“别漏”，第二轮重排追求“挑得准”。

第一轮混合检索通常会拿回：

- top 50
- top 100

这批候选里有用的很多，但噪声也不少。

所以第二轮用 cross-encoder re-ranker，对：

- query
- chunk

这个组合做更精细的相关性判断，再从里面选出 top 5 左右给 LLM。

它为什么更准？

因为 bi-encoder 的向量检索，是“query 自己编码、文档自己编码，再比较距离”。  
cross-encoder 则是“把 query 和文档一起看，再判断相关性”。

前者快，适合全库搜。  
后者准，适合小范围精排。

所以生产级做法通常是：

> 用快方法召回，用准方法重排。

---

## 6.11 ACL 权限过滤为什么是硬约束

原文这里的态度非常明确：

> 一次权限泄漏，就足以毁掉整个平台的信任。

企业内部知识系统不是公网上随便搜，它必须满足：

> 用户只能看到自己有权看的文档。

原文对比了两种做法：

### 1. 预过滤

在检索时就把用户权限条件带进去。

优点：

- 更安全。
- 更快。
- 不会先取回一堆不能看的文档再丢掉。

缺点：

- 如果权限特别严格，可选候选集会变小，检索质量可能略受影响。

### 2. 后过滤

先全量检索，再把无权限结果过滤掉。

实现简单，但问题也明显：

- 可能过滤完后结果不够。
- 内存里短暂出现了不该拿到的文档 ID。
- 某些合规要求下，这都不够安全。

所以原文明确偏向：

> 预过滤优先。

---

## 6.12 最后一步：怎么把检索结果组装进 prompt

retrieval 的输出不是直接给用户，而是先进入 `context assembly`。

这一步做的其实是 prompt 预算管理。

原文强调了几个关键点：

### 1. token 预算分配

上下文窗口不是全给检索结果用的，还要给：

- system prompt
- conversation history
- 最终回答输出

所以能塞多少 chunk，不是想塞多少就塞多少。

### 2. chunk 顺序

原文提到 `lost in the middle` 效应。

也就是说，模型对上下文开头和结尾往往更敏感，对中间内容相对不敏感。

所以不是简单按 rank 顺序平铺，而是要考虑摆放策略。

### 3. 去冗余和压缩

如果多个 chunk 内容高度重复，就没必要都塞进 prompt。

复杂查询时，甚至可以对长 chunk 先做提取式压缩，只保留和当前问题最相关的句子，减少 token 浪费。

这里的关键理解是：

> 检索返回 top-k 只是中间结果，真正进 prompt 的上下文还需要二次整理。

---

## 6.13 第 6 课的完整链路

把这一课串起来，完整流程大概是：

```text
用户提问
  ↓
查询理解：分类、改写、必要时做 HyDE
  ↓
并行检索：向量搜索 + BM25
  ↓
结果融合：RRF 或加权融合
  ↓
权限过滤
  ↓
re-rank 精排
  ↓
上下文组装：预算、去重、排序、压缩
  ↓
送入生成层
```

而这条链路背后依赖的存储层是：

```text
向量库 + 关键词索引 + 元数据库 + 语义缓存
```

这就是为什么生产级 RAG 的检索层，本质上是一个小型搜索平台。

---

## 6.14 这一课的核心结论

第 6 课学完，至少要建立这几个判断：

1. 生产级 RAG 的存储层通常不是一个库，而是向量库、关键词索引、元数据库、缓存的组合。
2. query understanding 是检索前置步骤，它决定“用户原话”如何变成“可搜准的查询”。
3. hybrid retrieval 的价值在于同时覆盖语义匹配和精确匹配。
4. re-rank 的作用不是扩大召回，而是把候选结果筛到真正适合喂给 LLM 的程度。
5. ACL 权限过滤和上下文组装，不是边角料，而是生产检索链路的核心环节。

---

# 第 7 课：为什么复杂问题要多轮搜索

## 7.1 这一课先记住一句话

> Agentic RAG 的本质，不是“加个 Agent 更高级”，而是让系统在复杂问题上学会先拆、再搜、再判断证据够不够。

到第 6 课为止，我们讲的主流程基本都是：

```text
提问 → 检索一次 → 生成答案
```

这就是 `single-shot RAG`。

原文明确说，这种方式能处理企业里大约 60% 到 70% 的问题，剩下那一批更难的问题，单次检索通常不够。

它们常见的问题是：

- 需要多个文档拼起来才能答。
- 需要跨主题推理。
- 用户问题太抽象，第一次检索抓不到真正目标。

这时才需要 `Agentic RAG`。

---

## 7.2 什么是 Agentic RAG

原文对它的定义可以直接翻译成人话：

> 让一个 LLM 不只是“根据检索结果回答”，而是参与决定“下一步该怎么搜”。

普通 RAG 更像：

> 去资料库查一次，拿回结果，然后回答。

Agentic RAG 更像：

> 像研究员一样，先拆问题，再查资料，发现资料不够，就继续查，直到证据足够或者到达边界为止。

所以两者最核心的差别不是有没有模型，而是：

| 普通 RAG | Agentic RAG |
|---|---|
| 一次检索后直接回答 | 可以多轮检索 |
| 检索路径基本固定 | 会根据结果动态决定下一步 |
| 更适合简单问题 | 更适合多跳、分析、比较、时间变化问题 |

---

## 7.3 不是所有问题都值得上 Agent

原文这里强调得非常明确：

> Not every query needs an agent.

为什么？

因为 agent 带来的不是只有“更强”，还有：

- 1.5 到 3 倍延迟增加。
- 2 到 5 倍成本增加。
- 更复杂的失败模式。

所以生产系统不能默认所有问题都走 agent。

真正适合单次检索的问题通常是：

- 事实查找题
- 简单 how-to
- 明确的错误码或标识符
- 单主题问题

适合 agentic 路径的问题通常是：

- 比较题
- 多跳题
- 时间变化题
- 原始提问太抽象的问题

原文举的例子很典型：

单次检索更适合：

- `What is the endpoint for the user service?`
- `ErrorCode 4032`
- `How to set up local dev for payments service?`

Agentic 更适合：

- `Compare auth v1 vs v2 and what changed after Q3`
- `Why is checkout slow?`
- `How does payments talk to user service and what happens when user service is down?`

所以这里真正的工程判断是：

> 让 agent 只处理“普通检索明显不够”的那部分高价值复杂问题。

---

## 7.4 系统怎么判断要不要进入 Agentic 路径

原文给的思路是先做一个轻量分类器。

它会看几类信号：

- 问题复杂度
- 检索置信度
- 问题长度
- 是否包含比较词
- 是否包含时间变化词

例如这些表达通常是强信号：

- `compare`
- `difference between`
- `trade-offs`
- `changed since`
- `before and after`
- `history of`

如果分类器很有把握地判断问题复杂，就直接走 agentic 路径。

如果判断不够确定，原文建议的做法是更保守一点：

> 先走 single-shot，如果检索置信度低，再回退到 agentic。

这个策略很现实，因为它在质量、延迟、成本之间更平衡。

---

## 7.5 第一步：query decomposition 在做什么

复杂问题往往不是一个问题，而是多个子问题缠在一起。

例如：

> payments service 怎么和 user service 做鉴权，Q3 auth migration 之后又改了什么？

这个问题至少包含三层：

1. 原本的鉴权机制是什么。
2. Q3 migration 改了什么。
3. payments service 在 migration 后具体变了什么。

如果不拆开，单次检索很可能抓不全。

所以 `query decomposition` 的作用就是：

> 把一个复杂大问题，拆成 2 到 5 个可以独立检索的小问题。

拆完之后，每个子问题都可以走完整检索链路。最后再把结果合并。

你可以把这一步理解成：

> 先把“研究课题”拆成几个可以分别查资料的小任务。

---

## 7.6 第二步：iterative retrieval 为什么重要

拆子问题只是第一层。更关键的是：

> 第一次检索拿回来的结果，可能还是“不够答”。

原文第 12.3 节的重点就是 `iterative retrieval with self-reflection`。

流程是：

1. 先搜一次。
2. 让 agent 判断：这些资料够回答吗？
3. 如果不够，再明确缺什么。
4. 根据缺口重新搜一次。
5. 最多循环几轮。

这里最重要的不是“多搜一次”，而是：

> 系统要能意识到自己“还缺什么信息”。

例如它第一次找到了 auth 文档，但发现那是旧版本；这时它需要继续搜“迁移之后的 auth 变更文档”。

这类能力，普通 single-shot RAG 是没有的，因为它不会反思第一次检索结果是不是足够。

---

## 7.7 multi-source routing：不是每次都搜全库

原文第 12.4 节讲了一个很实用的优化：

> 复杂问题不是每次都要在全量 1000 万 chunk 上盲搜。

系统可以先判断问题类型，再优先搜更合适的来源。

例如：

| 问题类型 | 更优先的来源 |
|---|---|
| API reference | GitHub README、Confluence API 文档 |
| 事故调查 | Slack thread、runbook |
| 架构决策 | RFC、设计文档、ADR |
| 安装配置 | Runbook、README |

这样做的好处有三个：

1. 缩小搜索空间。
2. 提高精度。
3. 降低检索延迟。

所以 agent 不只是“多搜”，而是：

> 更有策略地决定“去哪里搜”。

---

## 7.8 MCP 工具调用为什么成了 Agentic RAG 的核心

原文第 12.5 节开始把重点放到工具调用上。

核心思想是：

> Agent 不应该什么都靠语言猜，而应该调用合适的工具拿数据。

原文这里的工具包括：

- `search-vector`
- `search-keyword`
- `search-code`
- `query-metadata`
- `calculate`
- `fetch-document`

你可以看到，不同工具负责不同问题：

- 概念问题适合语义检索。
- 错误码适合关键词检索。
- 代码问题适合 code search。
- “谁负责这个服务”适合查结构化元数据。

这说明 Agentic RAG 的关键，不是“让模型自己更会想”，而是：

> 让模型知道什么时候该用哪把工具。

---

## 7.9 为什么 MCP 比自定义工具接口更适合生产

原文给了三个理由：

### 1. 模型可移植

工具写一次，可以给不同模型用，不用每换一个模型就重写工具格式。

### 2. 安全

每个工具都可以独立做认证和授权。

例如 code search 工具可以先检查用户有没有权限看这个 repo，再决定是否返回结果。

### 3. 能力协商

Agent 启动时可以知道：

- 当前有哪些工具可用。
- 每个工具需要什么输入。
- 某个工具挂了时能不能跳过。

所以 MCP 在这里的价值，不只是“协议统一”，更是：

> 把 Agent 的工具使用做成可控、可授权、可迁移的生产能力。

---

## 7.10 一个完整的 Agentic 例子是什么样

原文给了一个很典型的两轮例子：

问题：

> payments service 怎么处理 retries，Q3 migration 之后又改了什么？

### 第 1 轮

先拆成两个子问题：

- `payments service retry strategy`
- `Q3 migration changes payments`

然后分别搜。

结果发现：

- retry strategy 相关资料很充分。
- Q3 migration 相关结果太泛，不够具体。

所以 agent 得出结论：

> 当前证据不足，需要继续搜更具体的迁移变化。

### 第 2 轮

再做更精确的搜索：

- `Q3 migration payments retry`
- `payments Q3 breaking changes`

这次找到了：

- `max_retries` 从 5 改到 3
- timeout 从 10s 改到 5s
- jitter 算法变化

于是 agent 才进入最终生成。

这个例子最重要的不是答案内容，而是过程：

> agent 会根据“第一次搜得不够具体”这个判断，主动发起第二轮更精确的检索。

这就是 Agentic RAG 和普通 RAG 的根本区别。

---

## 7.11 为什么必须给 Agent 上护栏

原文第 12.6 节态度非常明确：

> 没有 guardrails 的 Agentic RAG，就是一个随时可能失控的成本炸弹。

为什么？

因为 agent 如果一直搜不到有效结果，可能会：

- 一直重复检索
- 不断调用工具
- 吃掉大量 token
- 把延迟拖得非常长

所以生产里要给它设硬边界，例如：

- 最大迭代次数
- 最大工具调用次数
- 最大 token 预算
- 最大 wall-clock 时间
- 单次查询成本上限

还要有 circuit breaker：

> 如果发现自己在打转，或者新一轮结果根本没变好，就及时停止，退回到 best-effort answer。

这一点非常关键。因为在生产里，通常是：

> 一个稍微差一点但及时返回的答案，比一个无限追求完美但迟迟不返回的答案更有价值。

---

## 7.12 第 7 课的完整链路

把这一课内容串起来，Agentic RAG 的主流程大概是：

```text
用户提问
  ↓
复杂度分类
  ↓
简单问题 → single-shot RAG
复杂问题 → 进入 agentic 路径
  ↓
拆解子问题
  ↓
按需要调用检索 / 代码 / 元数据等工具
  ↓
判断当前证据是否足够
  ↓
不够就继续 refine query 再搜
  ↓
达到充分性或达到护栏上限
  ↓
生成最终答案并附引用
```

所以 Agentic RAG 不是在替代 RAG，而是在扩展它：

> 把“一次检索”升级成“可反思、可迭代、可用工具的检索决策过程”。

---

## 7.13 这一课的核心结论

第 7 课学完，至少要建立这几个判断：

1. Agentic RAG 只适合复杂问题，不能默认所有查询都走它。
2. query decomposition 的作用，是把复杂问题拆成可独立检索的子问题。
3. iterative retrieval 的价值，在于系统能识别“证据还不够”，并继续搜缺失信息。
4. MCP 工具调用是 Agentic RAG 的核心基础设施，因为复杂问题往往需要多种数据获取方式。
5. 没有迭代上限、成本上限和超时保护的 Agent，会在生产里迅速失控。

---

# 第 8 课：答案是怎么生成出来又怎么避免乱编的

## 8.1 这一课先记住一句话

> 检索只是把资料找回来，真正把“资料”变成“可信答案”的，是生成层和它后面的一整套防幻觉约束。

很多人以为 RAG 只要检索做好，最后让模型回答一下就结束了。

原文第 13-14 节讲的恰恰是：

> 就算资料已经找对了，如果生成层设计不好，模型仍然可能说得不稳、说得太慢、说得不带引用，甚至说错。

所以这一课讲的是后半段：

- 怎么选模型
- 怎么组 prompt
- 怎么流式返回
- 怎么压上下文预算
- 怎么强制引用
- 怎么做验证和护栏

---

## 8.2 生成层到底在做什么

生成层的任务不是单纯“调一下 LLM API”，而是把检索层的结果组织成用户最终看到的答案。

它至少要负责这些事：

1. 选择合适的模型 tier。
2. 组装 system prompt、上下文、历史对话。
3. 控制回答格式。
4. 强制 grounding 和 citation。
5. 把回答流式返回给前端。
6. 在回答后做校验和护栏检查。

一句话：

> 生成层负责把“检索结果”加工成“可信、可读、可交付的回答”。

---

## 8.3 为什么模型路由在生成阶段依然重要

原文一开头就重复了一个前面出现过的原则：

> 不是每个问题都值得用最贵的模型。

在生成层，这个原则依然成立，而且影响更直接，因为：

- 生成 token 本身最烧钱。
- 大模型 TTFT 更高。
- 复杂模型的吞吐更低。

原文给的思路是分 tier：

| 层级 | 更适合什么问题 |
|---|---|
| Fast | 简单事实题、定义题、单文档回答 |
| Standard | how-to、中等推理、多 chunk 综合 |
| Complex | 多跳分析、比较、复杂综合 |

所以这里真正的系统设计思想是：

> 检索层先判断问题复杂度，生成层再根据复杂度把问题送去最合适的模型。

原文还强调一个很现实的策略：

> 如果不确定，宁可往上路由，也不要往下路由。

因为模型太强，最多多花一点钱；模型太弱，可能直接给出不可靠答案，损失的是信任。

---

## 8.4 生成服务挂了怎么办

这一节很容易被忽略，但原文专门讲了 fallback chain。

意思是：

> 就算主要大模型服务挂了，系统也不应该直接报 500。

比较合理的降级路径通常是：

1. 先换同一层级的备用供应商。
2. 再不行就切到自托管模型。
3. 如果连自托管也不可用，至少返回“我找到了这些相关文档链接”。

这件事的重要性在于：

> 在最坏情况下，系统至少还能退化成一个搜索引擎，而不是完全不可用。

对于工程师工具来说，这通常比空白错误页强得多。

---

## 8.5 prompt 为什么是整个平台最关键的一段文本

原文有一句很重的话：

> The system prompt is the most critical piece of text in the entire platform.

为什么？

因为它直接决定：

- 模型是不是只基于检索上下文回答。
- 是不是必须带引用。
- 证据不够时会不会承认不确定。
- 回答格式是否一致。

原文的 prompt 规则可以压缩成 5 个关键词：

1. `grounding`
2. `citations`
3. `uncertainty`
4. `scope`
5. `format`

也就是说，系统 prompt 至少要明确告诉模型：

- 只能用检索到的上下文。
- 每个事实要能找到来源。
- 不确定就说不确定。
- 不要拿训练记忆来补空白。
- 输出格式要稳定。

这不是“写提示词小技巧”，而是：

> 用 prompt 先把模型行为边界画出来。

---

## 8.6 prompt 不是一份写死的文本

原文还强调两点很实用的工程实践：

### 1. prompt 要版本化

不能把 prompt 散落在代码里随便改。

因为 prompt 一改，回答风格、引用行为、幻觉边界都可能变。  
所以它应该像配置一样版本管理，并且能做 A/B test。

### 2. prompt 要按问题类型动态组装

不同问题的最佳 prompt 其实不一样：

- 事实题更强调简洁。
- how-to 更强调步骤化。
- 分析题更强调 trade-off 和 nuance。

这说明生成层不是“一个模板打天下”，而是：

> 根据问题类型和上下文动态组装 prompt。

---

## 8.7 为什么 streaming 很重要

原文第 13.3 节讲的是用户体验层面最明显的一个点：

> 用户不应该盯着空白屏幕等 2 秒。

这就是为什么生成层通常会做 streaming。

它的核心价值不是让总时长变短，而是：

> 让用户更早看到第一批 token，降低感知延迟。

原文给的典型时间分解是：

- 查询理解几十到几百毫秒
- 检索和重排一两百毫秒
- prompt 组装几毫秒
- LLM TTFT 几百毫秒

所以最终用户通常能在不到 1 秒多一点看到第一批内容，即使完整答案还要更久。

这对体验影响很大，因为：

> 先看到回答开始生成，用户会觉得系统是“活着的”。

---

## 8.8 为什么 citation 在流式输出里更麻烦

引用不是普通文字，它有结构。

原文提到的一个实际问题是：

> 模型输出 `[Source 1]` 时，可能不是一次性吐完整，而是拆成多个 token。

如果前端直接逐 token 生渲染，就可能出现：

- 引用标记半截显示
- 引用还没闭合
- URL 还没解析完

所以后端通常要对 citation marker 做一点 buffering：

> 等完整引用标记形成后，再安全地发给前端。

这说明生成层不只是“转发 token”，还要对特殊结构做协议级处理。

---

## 8.9 上下文窗口为什么要精细预算

原文第 13.4 节强调：

> context window 是共享资源，不是检索内容一个人独占。

因为在一次回答里，token 预算要分给多个部分：

- system prompt
- retrieved context
- conversation history
- 最终输出

所以你不能简单地说“搜到 15 个 chunk 就全塞进去”。

还要考虑：

### 1. 历史对话

多轮对话时，最近 2 到 3 轮通常值得保留。更早的历史则可以先总结。

### 2. 大上下文模型只给少数复杂问题

原文强调，200K 这种大窗口模型成本很高，所以只给真正需要积累大量上下文的复杂问题用。

### 3. 上下文管理也是成本控制

塞得太多不只是慢，还会直接推高生成费用。

所以这里真正的系统思想是：

> 上下文窗口不是越大越随便用，而是要像预算一样分配。

---

## 8.10 自托管生成模型在这个位置上意味着什么

原文第 13.5 节讲的是一个现实问题：

> 你不一定非得完全依赖 API 提供商。

生成层、embedding、re-rank，其实都可以有开源自托管替代。

这一节你不用背模型名，先记住三件事：

1. 自托管的意义主要在大规模成本、数据主权、定制能力。
2. API 的意义主要在接入快、运维轻。
3. 是否自托管，不是理念问题，而是规模和约束问题。

所以在生成层这里，原文不是在说“API 不好”或“开源更好”，而是在说：

> 当流量、合规、成本到了一定规模，自托管会变成合理选项。

---

## 8.11 为什么幻觉控制必须分层做

原文第 14 节的核心态度非常明确：

> 内部知识助手里，幻觉比“不知道”更危险。

因为工程师可能会拿答案直接去改系统、发版、处理事故。

所以防幻觉不能只靠一句 prompt，而要多层同时做。

原文这套防线可以概括成：

1. 检索 grounding
2. 置信度分级
3. 引用强制
4. 后验验证
5. NLI / 实体校验 / 敏感信息过滤
6. 结构化输出

你可以把它看成：

> 不相信模型一次就一定说对，而是让多个环节共同约束它。

---

## 8.12 grounding 和置信度分级在做什么

第一层防线是最基础的：

> 明确要求模型只能依据检索上下文回答。

但原文也很诚实地说，这还不够。模型还是可能说出“听起来合理但上下文里并没明确写”的东西。

所以还要加第二层：

> 根据检索结果质量，先估计这次回答到底有多可信。

原文给的是三档模式：

### 高置信度

正常生成完整答案。

### 中等置信度

生成答案，但要明显带保留语气，例如：

> Based on what I found...

并提醒用户最好再核对某个团队或来源。

### 低置信度

直接 abstain，不正常回答，只给最接近的文档链接或建议去哪里继续确认。

这层非常重要，因为它解决的是：

> 不是所有问题都该被同样自信地回答。

---

## 8.13 citation 为什么必须被强制，而不是靠模型自觉

原文把 citation enforcement 分成三层：

### 1. Prompt 层

先要求模型必须带 inline citation。

### 2. Structured output 层

让模型按固定 schema 输出：

- 答案正文
- 引用列表
- 置信度
- 是否需要进一步验证

这样可以减少“这次忘了引用”的随机性。

### 3. Post-generation verification 层

在答案生成后，再检查：

> 这个回答引用的 source，真的支持它刚刚说的那句话吗？

这一层特别关键，因为模型有时会出现“引用了，但引用得不对”的问题。

所以这里真正的原则是：

> citation 不只是格式问题，它是事实可追溯性的技术保证。

---

## 8.14 NLI、实体校验和结构化输出分别在补什么洞

原文后面又加了几层更细的保护：

### 1. NLI 检查

把回答里的 claim 和检索上下文做自然语言推断：

- `entailed`：上下文支持
- `neutral`：上下文没说
- `contradicted`：上下文相反

如果是 `neutral` 或 `contradicted`，就说明回答可能有问题。

### 2. 实体 grounding

检查回答里提到的服务名、接口名、配置项，是不是真的出现在上下文或更大语料里。

如果模型突然说出一个根本不存在的 `AuthServiceX`，这就是危险信号。

### 3. 安全过滤

过滤掉可能泄露的敏感信息，例如：

- API key
- 密码
- 内网 IP

### 4. 结构化输出

强制模型按 JSON 或 tool schema 输出。

这样做的价值不仅是格式统一，更重要的是：

> 让引用、置信度、后续建议这些结构信息变成“必须存在的字段”，而不是可有可无的自然语言习惯。

---

## 8.15 第 8 课的完整链路

把这一课内容串起来，生成层大概是这样：

```text
检索层返回高质量上下文
  ↓
按问题复杂度选择合适模型 tier
  ↓
动态组装 system prompt + context + history
  ↓
流式生成答案
  ↓
强制输出引用与结构化字段
  ↓
做置信度判断、citation 验证、NLI 检查、实体校验与安全过滤
  ↓
返回最终答案
```

所以生成层不是“模型说什么就发什么”，而是：

> 一条受约束、可验证、可降级的答案生成流水线。

---

## 8.16 这一课的核心结论

第 8 课学完，至少要建立这几个判断：

1. 生成层不仅负责调用模型，还负责路由、prompt 组装、streaming 和后验验证。
2. 模型路由在生成阶段依旧重要，因为它直接影响质量、TTFT 和成本。
3. grounding、citation、uncertainty、scope 这些 prompt 规则，是约束模型行为的第一道边界。
4. 幻觉控制必须分层做，不能只靠一句“请不要乱编”。
5. citation 验证、NLI、实体校验、结构化输出，都是把“看起来像真话”变成“更接近可证明真话”的手段。

---

# 第 9 课：怎么知道 RAG 真的变好了

## 9.1 这一课先记住一句话

> 如果你不持续测量 RAG 的质量，那你并不知道系统是在变好，还是只是“看起来还行”。

原文第 15-16 节的态度非常直接：

> “看 demo 感觉不错”不算指标。

这是因为 RAG 和普通软件不一样。普通接口只要返回结构对了，很多时候就算稳定；RAG 则可能：

- 返回了答案，但事实错了。
- 资料找对了，但模型总结错了。
- 答案正确，但引用不对。
- 对某类问题一直很差，但整体平均分被掩盖了。

所以这一课讲的是：

> 怎么把“感觉它挺聪明”变成“我能证明它确实更好了”。

---

## 9.2 为什么评估是生产要求，不是可选项

原文一上来就说：

> Evaluation is a production requirement, not a nice-to-have.

意思是：

> 评估不是锦上添花，而是生产系统的必需部分。

为什么这么重？

因为 RAG 的质量退化很多时候不会像普通系统那样直接报错，它更可能是：

- 召回慢慢变差。
- 某类问题开始答偏。
- 某次 prompt 改动让引用行为退步。
- embedding 升级后检索 subtly 变差。

这些问题如果没有评估体系，往往只能等用户抱怨后才发现。

所以原文的核心思想是：

> 对 RAG 来说，不测量就等于盲飞。

---

## 9.3 离线评估在做什么

离线评估可以先理解成：

> 在一批事先整理好的标准问题上，重复跑系统，看它答得怎么样。

原文这里的核心是 `golden dataset`。

也就是一组经过人工整理的高质量问题集合，通常包含：

- 问题本身
- 期望答案里应该出现什么
- 期望来源文档是什么
- 问题类别和难度

原文给的分布思路很有代表性：

- 简单事实题
- how-to
- 分析题
- multi-hop 题

这说明 golden dataset 不是随便凑一些问题，而是：

> 要覆盖你系统真正关心的几类使用场景。

---

## 9.4 “正确”不是一个单一概念

这是原文里一个很重要但容易被忽略的点：

> correct 的含义，会随着问题类型不同而不同。

例如：

### 事实题

更看重：

- 是否精确
- 是否说对具体字段、错误码、接口

### how-to

更看重：

- 步骤是否完整
- 有没有漏掉关键前提

### 分析题

更看重：

- 推理是否连贯
- trade-off 是否讲对

所以如果你只看一个总体平均分，很容易被掩盖问题。

例如系统在事实题上很好，但在分析题上很差；总分看起来还不错，但真实使用体验已经有明显缺口。

这一节真正要记住的是：

> 评估必须按问题类型拆开看。

---

## 9.5 离线评估要看哪些指标

原文提了两类指标：

### 一类：偏答案质量

例如：

- `faithfulness`
- `answer relevance`
- `context relevance`
- `context recall`

这些问题分别在看：

- 回答是不是只基于上下文
- 回答是不是对题
- 取回来的 chunk 是否相关
- 取回来的 chunk 是否包含了真正需要的信息

### 一类：偏检索质量

例如：

- `Recall@5`
- `Recall@10`
- `MRR`
- `NDCG@10`

这一类不直接评价自然语言好不好，而是评价：

> 检索层到底有没有把正确资料排到前面。

这一点特别重要，因为原文一直强调：

> 生成差和检索差是两种完全不同的问题。

所以评估时，必须把它们拆开。

---

## 9.6 为什么 golden dataset 要像 CI 一样跑

原文把这一点说得非常像工程常识：

> 只要你改了会影响质量的东西，就应该自动重跑 golden dataset。

例如这些改动都应该触发回归评估：

- chunking 逻辑变化
- embedding 模型变化
- 检索参数变化
- re-rank 模型变化
- system prompt 变化

如果关键指标下降超过阈值，就应该：

- 阻断发布
- 提醒团队

这就是原文说的：

> 这是 RAG 版的 CI 测试套件。

换句话说：

> 评估不是做报告，而是要进入发布流程。

---

## 9.7 在线评估为什么必不可少

离线评估再精心，也无法覆盖真实流量的复杂度。

原文说得很清楚：

> 生产里的问题更脏、更乱、更含糊。

所以还必须做 online evaluation。

在线评估主要看三种来源：

### 1. 显式用户反馈

例如：

- 点赞 / 点踩
- 用户手写纠错
- 复制答案行为

### 2. 隐式行为信号

例如：

- 很快又问一次，说明上一个答案可能不够好
- 用户直接离开
- 用户改写问题重新问

### 3. LLM-as-judge

抽样生产流量，让另一个模型去评价：

- 正确性
- 完整性
- 引用质量
- 清晰度

这一层的意义在于：

> 你不可能全靠用户手动打分，所以需要自动化的“近似裁判”。

---

## 9.8 LLM-as-judge 为什么有用，但不能盲信

原文把它当成生产反馈回路的一部分，而不是最终真理。

它有几个明显价值：

- 可以大规模抽样生产查询。
- 可以比人工更快给出趋势判断。
- 可以帮助发现哪些回答经常 hallucinate。

但它也有边界：

- 它本身也是模型，不是绝对真值。
- 样本比例越高，成本越大。
- 判断标准写不好，也会带偏系统。

所以更合理的理解是：

> LLM-as-judge 是一种自动化质量信号，不是替代人类审查的万能裁判。

原文这里甚至提醒了成本问题：如果月查询量很大，哪怕只抽 5%，累计评估量都很可观。

这说明：

> 评估系统本身也是有成本的，需要像主系统一样做预算。

---

## 9.9 A/B test 和持续改进闭环在做什么

原文第 15.3 节真正想讲的，不只是“测完打分”，而是：

> 测量结果要能反过来驱动系统改进。

这条改进链路包括：

1. 低分答案每周复盘。
2. 用户纠错进入 golden dataset。
3. 检索失败触发文档和 chunking 审计。
4. 模型升级前先跑完整基准。
5. 点踩样本反向影响排序和 chunk 质量判断。

这里最值得记的一句话是：

> 很多团队会收集反馈，但不会把反馈真正接回检索和生成参数里。

而原文认为，这恰恰是最有价值、也是最容易被漏掉的部分。

---

## 9.10 为什么一定要把 retrieval 和 generation 分开观测

这是原文反复强调的一个核心原则：

> 一个坏答案，可能是检索错了，也可能是生成错了。

如果你只看最终用户评分，就不知道根因在哪里。

举个最典型的区别：

### 情况一：检索错了

- top-10 里根本没有正确资料

那问题通常在：

- chunking
- embedding
- 检索参数
- 索引健康度

### 情况二：检索对了

- 正确资料已经在上下文里

但回答仍然错了，那问题通常在：

- prompt
- 模型选择
- guardrails
- citation 生成或验证

所以原文的结论非常工程化：

> 质量诊断必须拆层，先看 retrieval，再看 generation。

---

## 9.11 为什么可观测性不是“看个日志”就够了

第 16 节讲 observability，重点是：

> 当用户说“这个答案不对”时，你必须能定位到底是哪一层出了问题。

原文这里推的是全链路 tracing。

一条 query 的 trace 应该能看到：

- query understanding 花了多久
- vector search 拿回多少结果
- BM25 拿回多少结果
- RRF 合并后有多少
- ACL 过滤掉了多少
- re-rank 分数怎么样
- 用了哪个模型
- 输入 / 输出 token 是多少
- 花了多少钱
- guardrails 是否通过

这类 trace 的价值是：

> 你不只是知道“系统慢了”或“系统答错了”，而是知道慢在第几层、错可能错在第几层。

---

## 9.12 监控面板应该盯哪些指标

原文给了一整套 dashboard 指标。你现在不必背全部，但要知道它们大致分几类：

### 延迟类

- 端到端延迟
- TTFT
- retrieval latency
- rerank latency

### 质量类

- retrieval relevance
- thumbs up 比例
- hallucination rate

### 成本类

- token usage
- cost per query

### 健康类

- cache hit rate
- error rate
- embedding pipeline lag

这里要建立的感觉是：

> 一个成熟 RAG 平台的 dashboard，不只是看 CPU 和 QPS，而是同时看质量、成本、延迟和数据新鲜度。

---

## 9.13 成本监控和 embedding drift 为什么也属于 observability

原文最后两节补了两个很实际的问题：

### 1. 成本异常

例如：

- agent loop 失控
- prompt 回归导致输出变长
- cache 失效让生成量暴涨

这些都可能让成本突然翻倍。

所以生产里要能做到：

- 每条 query 有成本记录
- 有短周期成本异常报警
- 必要时自动降级到更便宜模型

### 2. embedding drift

这是很多人不容易第一时间想到的问题：

> 即使你没主动改 embedding 模型，供应商侧变化也可能让向量空间悄悄漂移。

这时系统不会直接报错，但检索质量会慢慢变差。

所以原文建议：

- 维护一组 canonical queries
- 定期重新算 embedding
- 和历史参考向量比较
- 漂移超过阈值就报警

这说明 observability 不只是“监控当前请求”，还包括：

> 监控模型和数据分布是否在悄悄变化。

---

## 9.14 第 9 课的完整链路

把这一课串起来，评估和监控体系大概是：

```text
离线 golden dataset
  ↓
每次关键改动自动回归评估
  ↓
生产流量中的用户反馈 + LLM-as-judge 抽样
  ↓
质量问题进入周度复盘
  ↓
反馈反哺 chunking / retrieval / prompt / model routing
  ↓
同时用 tracing + dashboard + anomaly alert 持续监控延迟、质量、成本、漂移
```

所以评估体系不是单独的一页报表，而是：

> 连接“开发、发布、线上反馈、持续优化”的质量闭环。

---

## 9.15 这一课的核心结论

第 9 课学完，至少要建立这几个判断：

1. 离线评估和在线评估缺一不可，前者控回归，后者反映真实使用。
2. golden dataset 是 RAG 的质量基准，应该像 CI 一样自动运行。
3. retrieval 和 generation 必须拆开测量，否则坏答案无法定位根因。
4. LLM-as-judge、用户反馈、A/B test 要进入持续改进闭环，而不是停留在报告层。
5. observability 在 RAG 里不仅是性能监控，还包括质量、成本、新鲜度和 embedding drift。

---

# 第 10 课：一套 RAG 系统真正上线前还缺什么

## 10.1 这一课先记住一句话

> 一个能演示的 RAG 系统，离真正上线，通常还差“稳态运行能力”这一整层东西。

前 9 课讲的是：

- 文档怎么进来
- chunk 怎么切
- embedding 怎么做
- 检索怎么找
- agent 怎么多轮搜索
- 生成怎么防幻觉
- 评估和监控怎么闭环

但这些还不等于“可以上线”。

原文第 17-21 节补的是最后一公里：

- 在高峰流量下会不会崩
- 某个依赖挂了会不会整条链路断掉
- 成本会不会一夜失控
- 会不会泄露权限或秘密
- 出问题时有没有清单知道先查什么

所以第 10 课的主题是：

> 怎么把“能跑的系统”变成“能长期承受真实生产流量的系统”。

---

## 10.2 生产架构真正关注的不是功能，而是退化能力

原文第 17 节一开始就在讲扩容、降级和瓶颈，而不是功能特性。

这说明生产架构最核心的不是：

> 功能是否齐全

而是：

> 当流量变大、依赖变慢、某个组件失效时，系统能不能继续提供“至少还有用”的响应。

这和 demo 的最大差别就在这里：

- demo 追求把最好效果跑出来
- 生产追求在坏情况下也别完全死掉

原文最重要的思想可以压成一句话：

> 用户应该永远尽量拿到某种有用结果，而不是空白错误页。

---

## 10.3 扩容和 load shedding 在解决什么问题

原文第 17.1 节讲的是 scaling strategy。

不同组件扩容信号不同：

- 向量库看向量量级和分片压力
- Elasticsearch 看索引体积
- embedding worker 看 Kafka backlog
- LLM inference pool 看并发队列深度
- API server 看 QPS 和 CPU

但更关键的是 `load shedding`，也就是过载时主动退化。

原文给的退化顺序非常有代表性：

1. 先关掉 agentic RAG
2. 再减少 re-rank 范围
3. 再强制所有请求走最便宜模型
4. 再停掉 LLM-as-judge
5. 最后只服务缓存命中，其他给降级响应

这套设计背后的思想是：

> 在压力增大时，优先牺牲“质量增强层”，保住“基础可用层”。

也就是说，系统宁可先变笨一点，也别先死掉。

---

## 10.4 failure handling 和 degradation ladder 为什么是上线必需品

原文第 17.2 节讲的是故障处理。

这里最重要的不是“有没有重试”，而是：

> 不同依赖坏了之后，系统分别退化到哪一层。

例如：

- LLM provider 挂了，可以切备用 provider
- Qdrant 挂了，可以从副本读
- Elasticsearch 挂了，可以退成 vector-only
- Redis 挂了，可以跳过缓存继续全链路
- embedding API 挂了，可以让新文档先排队，老索引继续服务

原文甚至给了一个很实用的 degradation ladder：

- Level 0：完整链路
- Level 1：跳过 re-rank
- Level 2：只用 BM25
- Level 3：只返回缓存
- Level 4：静态 fallback

这层设计的重点是：

> 不是“有没有故障”，而是“故障发生后还能剩下几成功能”。

---

## 10.5 multi-tenant isolation、rate limiting、backpressure 在防什么

原文第 17.3-17.4 节讲的是多租户和流量治理。

在企业内部平台里，不同团队共用同一套系统，最怕三件事：

1. 数据串租户
2. 一个团队把资源吃光
3. 后台大规模处理拖慢前台查询

所以这里有三层保护：

### 1. 数据隔离

每个租户至少要有严格的 `tenant_id` 过滤或命名空间隔离，确保不会跨团队泄露内容。

### 2. 成本隔离

每个团队的 token 和花费要单独记账、单独控预算。

### 3. 性能隔离

要靠 per-tenant rate limiting、priority queue、backpressure，防止：

- 某个团队突然爆量
- 某次批量入库
- 某个后台任务

把整个平台拖垮。

原文在这里真正想表达的是：

> 多租户平台上线前，隔离比吞吐更先决定信任。

---

## 10.6 bottleneck 和成本为什么总会互相缠在一起

原文第 17.5 和第 18 节连起来看，会发现一个现实规律：

> 大多数性能瓶颈，最后都会体现成成本问题；大多数成本问题，也常常从系统设计瓶颈里长出来。

原文举的典型瓶颈包括：

- LLM 高峰期排队
- ACL 复杂过滤导致检索变慢
- embedding backlog 让 freshness 超 SLA
- context 太长让答案漏信息
- re-rank 太慢
- 周一早上缓存冷启动

而成本部分又明确说：

> LLM inference 往往占总成本的 60% 到 85%

这意味着最大成本杠杆并不神秘，主要就两个：

### 1. 模型路由

让大多数简单问题走小模型，复杂问题才上大模型。

### 2. 语义缓存

避免重复问题重复调用 LLM。

所以第 18 节的核心不是“精确价格表”，而是：

> 成本控制的第一性问题，是减少不必要的大模型调用。

---

## 10.7 API 还是自托管，本质是在算哪笔账

原文第 18 节对 self-hosted vs API 的态度很务实。

它不是问：

> 哪个技术路线更高级？

而是问：

> 你的量级、合规和团队能力，决定哪笔账更划算？

通常：

### API 更合适

- 查询量还不大
- 不想养 GPU 运维
- 希望尽快落地

### 自托管更合适

- 查询量极大
- 对数据出网敏感
- 需要更强可控性
- 团队有 MLOps / infra 能力

所以这里真正要建立的判断是：

> 架构选型最后要回到流量规模、硬件成本、运维成本和合规约束上。

---

## 10.8 安全和治理为什么是红线

原文第 19 节的态度非常明确：

> 内部知识平台最怕的不是“偶尔答得差”，而是“把不该暴露的信息放大暴露”。

这里至少有三条硬线：

### 1. Prompt injection 防御

用户、文档或上下文里可能混进“忽略之前指令”之类内容。  
所以要有：

- 输入清洗
- 明确的指令层级
- 输出验证

### 2. PII 和 secret 处理

内部文档里可能包含：

- 邮箱
- 手机号
- API key
- token
- 密码

系统不能让这些内容因为“搜索方便”而更容易泄露。  
所以要在 ingestion 时标记，在 response 时再做 redaction。

### 3. 审计日志

必须能追踪：

- 谁问了什么
- 检索到了什么
- 生成了什么
- 用了什么模型和 prompt
- 用户反馈了什么

一句话：

> 对企业 RAG 来说，安全不是附加模块，而是平台级底座。

---

## 10.9 系统会怎么失败，为什么要提前假设它会失败

原文第 20 节非常有价值，因为它不是继续讲设计，而是诚实地讲：

> 这套系统在生产里会怎么坏。

原文列的典型 failure chain 很值得记：

### 1. 坏 chunking → 坏检索 → 自信错答

这是最经典的链式故障。

### 2. ACL bug → 数据泄漏

这个是最致命的信任事故。

### 3. embedding drift → 检索静悄悄变差

不会报错，但质量持续下降。

### 4. 缓存返回旧答案

系统“看起来很快”，但内容已经过期。

### 5. query rewriting 改坏问题

系统以为自己在帮忙，实际把用户问题带偏了。

### 6. agent loop 烧 token 不收敛

成本和延迟一起失控，答案却不见得更好。

这一节最重要的启发是：

> 不要只设计“理想流程”，要设计“坏链条是怎么形成的，以及怎么被发现”。

---

## 10.10 上线前为什么必须有 production readiness checklist

原文第 21 节其实是在做一件很工程的事：

> 把“上线前应该想清楚的事”从经验，变成结构化检查表。

它不是认证，而是 gap analysis，也就是找缺口。

原文把检查项分成 10 个维度：

1. `Ingestion and Chunking`
2. `Retrieval Quality`
3. `Generation and Hallucination Control`
4. `Agentic RAG and Tool Use`
5. `Evaluation and Feedback`
6. `Observability and Cost`
7. `Resilience and Production Hardening`
8. `Security and Governance`
9. `Data Quality and Freshness`
10. `Operational Readiness`

这里最重要的不是把 85 条都背下来，而是理解：

> 一套 RAG 系统要上线，不只是“回答质量过关”，而是这 10 个维度都不能有明显短板。

原文还给了一个很实用的评分思想：

- 120+：比较接近 production-ready
- 90-119：可以带着已知问题上线，但要限期补
- 60-89：还有明显风险
- 60 以下：还是原型期

你可以把这套 checklist 理解成：

> 用来回答“我们到底离上线还有多远”的工程尺子。

---

## 10.11 为什么这套 checklist 最终还是在回答同一个问题

把 17-21 节串起来，原文其实一直在回答一个总问题：

> 当系统离开 demo 环境，进入真实组织、真实流量、真实权限和真实事故场景时，它还能不能稳定地创造价值？

这也是第 10 课的总主题：

- 架构要能扩
- 流量要能控
- 故障要能退
- 成本要能算
- 安全要能守
- 失败要能看见
- 上线要有清单

如果这些没有补齐，前 9 课做得再漂亮，也只是一个“功能上成立”的系统，而不是“运营上成立”的平台。

---

## 10.12 第 10 课的核心结论

第 10 课学完，至少要建立这几个判断：

1. 生产级 RAG 的关键差别，在于它必须能降级、能隔离、能恢复，而不是只在理想路径上表现好。
2. LLM 推理通常是最大成本项，所以模型路由和缓存是最重要的成本杠杆。
3. 安全、权限、PII、secret redaction、审计日志，都是企业 RAG 的红线能力。
4. 系统真正的风险常常出现在 failure chain，而不是单个组件本身。
5. production readiness checklist 的作用，不是证明自己完美，而是系统化发现上线前的缺口。

# 总复习版：10 课一页总览

## 先用一句话概括整套课

> RAG 不是“给大模型接个向量库”，而是一条从文档摄取、切块、向量化、检索、生成、评估到生产治理的完整工程链路。

如果只记一句，就记这句。

---

## 总复习版：10 课压缩总结

### 第 1 课：RAG 是为了解决什么问题

- 企业知识太多、太散、更新太快，不能直接塞进大上下文。
- 关键词搜索只擅长字面匹配，不擅长理解意图。
- RAG 的基本形式是：先检索，再生成，并且答案要带来源。

### 第 2 课：生产级 RAG 的设计原则

- 检索质量决定答案上限，大模型不能凭空修正坏检索。
- chunk 质量比 chunk 数量更重要。
- 模型路由、拒答能力、评估体系、文档新鲜度，都是生产级必需项。

### 第 3 课：文档怎么进入系统

- ingestion 负责把不同来源文档变成统一、结构化、可检索的输入。
- 连接器不能只靠 webhook，要有 polling 和 reconciliation。
- 元数据、权限和去重，是 ingestion 的核心组成部分。

### 第 4 课：为什么 chunking 决定 RAG 上限

- chunk 太小会丢上下文，太大会稀释信号。
- 固定切块只是基线，生产里通常要递归切块、语义切块、多策略路由。
- 最终目标是得到“可独立检索、可独立理解”的知识单元。

### 第 5 课：embedding 为什么重要

- embedding 把文本映射到可比较语义相似度的向量空间。
- 文档入库和用户查询必须使用同一个 embedding 模型。
- embedding 升级本质上是一次向量数据迁移，不是改个配置而已。

### 第 6 课：检索层怎么工作

- 生产级检索层通常由向量库、关键词索引、元数据库、缓存共同组成。
- query understanding、hybrid retrieval、RRF、re-rank、ACL 过滤和 context assembly 是主链路。
- retrieval 的目标不是“搜到一些”，而是“搜到最适合喂给模型的证据”。

### 第 7 课：为什么复杂问题要多轮搜索

- single-shot RAG 只适合简单问题。
- Agentic RAG 适合多跳、比较、时序变化、抽象问题。
- query decomposition、iterative retrieval、MCP 工具调用和护栏上限是 agentic 路径的关键。

### 第 8 课：答案怎么生成又怎么避免乱编

- 生成层不仅负责调模型，还负责模型路由、prompt 组装、streaming、引用和后验验证。
- grounding、citation、uncertainty、scope 是 prompt 的核心约束。
- 防幻觉必须分层做：置信度分级、citation 校验、NLI、实体校验、结构化输出。

### 第 9 课：怎么知道系统真的变好了

- 没有评估就等于盲飞。
- 离线 golden dataset 用来控回归，在线反馈和 LLM-as-judge 用来反映真实使用。
- retrieval 和 generation 必须分开观测，质量、成本、延迟、漂移都要持续监控。

### 第 10 课：为什么能跑不等于能上线

- 生产系统必须能扩容、能降级、能隔离、能恢复。
- LLM 成本通常是最大头，模型路由和缓存是第一成本杠杆。
- 安全、权限、PII、secret redaction、审计日志和 readiness checklist 是上线红线。

---

## 总复习版：RAG 端到端主链路

把 10 课串起来，完整链路应该在你脑子里长成这样：

```text
外部知识源变化
  ↓
连接器发现更新
  ↓
解析正文、元数据、权限
  ↓
去重
  ↓
按文档类型切 chunk
  ↓
为 chunk 生成 embedding
  ↓
写入向量库 / 关键词索引 / 元数据库
  ↓
用户提问
  ↓
query understanding（分类、改写、必要时 HyDE）
  ↓
向量检索 + BM25 并行
  ↓
RRF / 加权融合
  ↓
ACL 过滤
  ↓
re-rank 精排
  ↓
context assembly
  ↓
模型路由
  ↓
生成答案 + 引用
  ↓
NLI / citation 校验 / 安全过滤
  ↓
返回答案
  ↓
收集反馈、评估、监控、持续优化
```

如果你能不用看材料，把这条链路完整讲出来，说明主线已经吃透了。

## 总复习版：你现在最该记住的 10 个判断

1. RAG 首先是检索问题，其次才是生成问题。
2. 长上下文不是大规模企业知识问答的根本解法。
3. chunking 是检索质量最关键的前置环节之一。
4. embedding 模型统一性是硬规则，不能文档和查询各用一套。
5. hybrid retrieval 几乎是生产系统默认配置，因为语义匹配和精确匹配互补。
6. re-rank 的任务不是扩大召回，而是把候选结果筛到足够准。
7. Agentic RAG 只适合复杂问题，不能默认全量开启。
8. grounding、citation、abstention 是可信 RAG 的基本底线。
9. 评估必须同时覆盖离线基准、在线反馈和全链路监控。
10. 真正的生产系统必须考虑降级、隔离、成本、安全和上线清单。

---

## 总复习版：最常见的 10 个误区

1. 以为 RAG 主要靠大模型强弱决定效果。
2. 以为文档随便切一下就行。
3. 以为所有文档都能用同一种 chunking 策略。
4. 以为换更强 embedding 模型只要改一行配置。
5. 以为向量检索可以替代关键词检索。
6. 以为 top-k 一拿回来就能直接丢给模型。
7. 以为复杂问题理所当然都该走 agent。
8. 以为 prompt 写一句“不要乱编”就够了。
9. 以为用户不投诉就说明系统质量没问题。
10. 以为功能跑通就等于可以上线。

## 总复习版：如果你自己做一套 RAG，优先顺序应该是什么

### 第 1 层：先让主链路成立

- 接通文档摄取
- 做基础 chunking
- 做 embedding
- 建向量检索和关键词检索
- 让回答能带来源

### 第 2 层：再把质量做好

- 多策略 chunking
- hybrid retrieval
- re-rank
- prompt 约束
- citation 校验

### 第 3 层：再把复杂问题做好

- query rewriting
- agentic 路径
- MCP 工具调用
- 复杂查询的 context compression

### 第 4 层：最后把生产化补齐

- golden dataset
- 在线反馈闭环
- tracing 和 dashboard
- 成本监控
- 权限和安全治理
- 降级与故障恢复

这条优先级本质上是在告诉你：

> 先保证“能搜到”，再保证“搜得准”，再保证“复杂情况也能答”，最后保证“线上长期跑得住”。

## 总复习版：如果拿这套内容去做系统设计，最值得先讲什么

如果以后你要把这套材料讲给别人听，或者拿去做系统设计题，建议优先讲这几个点：

1. 为什么需要 RAG，而不是长上下文或微调。
2. 为什么 chunking、embedding、hybrid retrieval 是质量核心。
3. 为什么企业场景必须做 ACL、citation 和 abstention。
4. 为什么复杂问题需要 agentic 路径，但不能全量默认开启。
5. 为什么质量评估、成本治理和 graceful degradation 决定系统能不能上线。

这 5 个点讲清楚，已经能覆盖大部分面试、设计评审和方案讨论。

## 总复习版：你现在应该形成的整体判断

学完这 10 课后，你对 RAG 的整体判断应该是：

> 一套企业级 RAG 平台，本质上是“文档数据系统 + 检索系统 + 生成系统 + 评估监控系统 + 生产治理系统”的组合。

再压缩一点：

> 先把知识变成高质量 chunk，再把 chunk 变成高质量检索结果，再把检索结果变成可信答案，最后靠评估和生产治理让整个系统长期可用。

如果这个判断已经稳定了，说明这份材料的主线你已经真正吃下来了。
